# 工作流节点审批组织展开关系

## 变更原因

当前工作流定义层已经落地 `tb_workflow_node_approver`，但仍有两个口径需要收敛：

- 节点审批人与流程节点是标准一对多关系，不再允许在 `approver_value` 中通过逗号拼接多个审批主体。
- 当 `approver_type=DEPT` 时，仅保存直接配置的组织节点还不够，还需要像用户组织展开、角色数据权限展开一样，额外保存向下展开后的当前生效组织范围。

如果继续保留“逗号拼接多个值”的写法，会让审批人配置的语义不稳定，也会让组织下级范围无法以结构化方式维护和重建。

## 变更内容

### 1. 审批人 direct 表口径收敛

- `tb_workflow_node_approver` 继续作为节点审批人的直接配置表。
- 一条 `tb_workflow_node_approver` 记录只允许对应一个审批主体。
- `approver_value` 只允许保存单个值，不允许再使用逗号拼接多个用户、角色或组织 ID。
- 一个审批节点如需配置多个审批主体，必须通过多条 `tb_workflow_node_approver` 记录表达。

### 2. 新增审批组织展开表

新增表：`tb_workflow_node_approver_dept_expand`

用途：

- 仅用于保存 `tb_workflow_node_approver.approver_type=DEPT` 的向下展开结果。
- direct 表承载“配置了哪个组织”。
- expand 表承载“该组织向下展开后当前覆盖了哪些有效节点”。

建议字段：

- `id`
- `approver_id`：来源审批人配置 ID，对应 `tb_workflow_node_approver.id`
- `node_id`：来源流程节点 ID
- `definition_id`：来源流程定义 ID
- `source_dept_id`
- `source_org_type`
- `source_post_type`
- `dept_id`
- `org_type`
- `post_type`
- `relation_type`：`SELF` / `DESCENDANT`
- `distance`
- `created_at`

### 3. 重建规则

工作流定义保存时：

- 新增流程定义后，针对本次保存的 `DEPT` 类型审批人，全量重建其展开记录。
- 修改流程定义时，先删除旧审批人与旧展开记录，再按新的 direct 配置重建 expand 记录。
- 删除流程定义时，同时删除该定义下审批人的展开记录。

组织树变更时：

- 组织新增、删除、移动、状态变化、类型变化、岗位类型变化后，需要重建受影响审批人的组织展开记录。
- 受影响审批人的判定规则与用户组织、角色数据权限的展开规则保持一致：
  - 统一基于 `tb_user_dept.path` 判断祖先链；
  - 命中祖先链上的 `DEPT` 类型审批人配置，需要重建其 expand 结果。

### 4. 当前有效节点口径

- `tb_workflow_node_approver_dept_expand` 仅保存当前有效组织节点。
- 来源组织已删除或停用时，旧 expand 记录在重建时应被清空，不再保留脏数据。
- 非岗位目标节点的 `post_type` 固定为空；岗位节点保留 `post_type`。

## 影响说明

- 设计影响：工作流定义层审批人配置从“允许单值或逗号拼接”收敛为“单记录单主体”。
- 数据库影响：新增迁移 SQL `sql/20260323.sql`，新增 `tb_workflow_node_approver_dept_expand` 并调整 `tb_workflow_node_approver.approver_value` 注释口径。
- 代码影响：
  - 工作流定义保存逻辑需要拒绝逗号拼接值。
  - 新增审批组织展开维护服务。
  - 组织管理变更后，需要同步重建受影响审批配置的展开关系。
- 测试影响：
  - 需要补充工作流审批组织展开服务测试。
  - 需要补充“拒绝逗号拼接审批人值”的服务测试。
