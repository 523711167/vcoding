# 流程定义保存原始流程 JSON

## 变更原因

当前流程定义新增、修改接口已经收敛为前端直接提交一份流程设计 JSON 字符串，由后端负责解析节点、审批人和连线再落库。

如果只保留解析后的结构化表数据，流程定义详情回显时前端还需要重新拼装流程图数据，联调成本高，也容易出现字段口径不一致。

## 变更内容

- `tb_workflow_definition` 新增 `workflow_json` 字段，保存前端提交的原始流程设计 JSON。
- 流程定义详情接口新增返回 `workFlowJson` 字段，前端可直接取值渲染。
- 流程定义新增、修改接口继续只接收 `workFlowJson`，后端统一解析后写入：
  - `tb_workflow_node`
  - `tb_workflow_node_approver`
  - `tb_workflow_transition`
- `tb_workflow_node` 移除 `code` 字段，前端节点 `id` 仅在保存时用于解析节点关系，不再单独存表。

## 影响说明

- 数据库影响：
  - 更新迁移 SQL `sql/20260323.sql`，新增 `tb_workflow_definition.workflow_json`。
  - 同步删除 `tb_workflow_node.code` 及其唯一索引。
- 接口影响：
  - `/sys/workflow-definition/create`
  - `/sys/workflow-definition/update`
  - `/sys/workflow-definition/detail`
- 代码影响：
  - 流程定义服务改为解析前端流程 JSON，再做节点、审批人、连线的校验和持久化。
  - 流程定义详情返回原始 JSON，同时保留结构化节点和连线明细。
- 兼容性说明：
  - 旧的 `nodes/transitions` 复杂入参不再保留。
  - 前端若仍依赖详情接口内结构化节点明细，当前仍可继续使用；若只需要流程图渲染，可直接使用 `workFlowJson`。
