# 查询箱“创建人组织 + 查看人最大数据权限”设计

## 1. 变更背景

查询箱可见性调整为基于“创建人组织范围”和“查看人角色数据权限范围”的组合判断，不再是简单同组织交集。

## 2. 业务口径冻结（按当前确认）

### 2.1 基础对象

- 创建人记为 `A`（申请单 `ba.applicant_id`）。
- 查看人记为 `B`（当前登录用户 `currentUserId`）。

### 2.2 创建人组织范围 `A_ALL_DEPTS`

1. 先取创建人 `A` 的主组织 + 副组织（`tb_user_dept_rel`）。
2. 对主副组织都做展开（`tb_user_dept_rel_expand`）。
3. 得到 `A_ALL_DEPTS`（创建人“所有组织范围”）。

### 2.3 查看人最大数据权限范围 `B_SCOPE_DEPTS`

查看人 `B` 可能有多个角色，按角色 `data_scope` 组合得到最终范围：

1. 若同时存在 `ALL` 与其他范围：取 `ALL`。
2. 若同时存在 `CURRENT_DEPT` 与 `CURRENT_AND_CHILD_DEPT`：取 `CURRENT_AND_CHILD_DEPT`。
3. 若 `CUSTOM_DEPT` 与其他组织范围组合：按交集收敛（例如 `CUSTOM_DEPT + CURRENT_DEPT`）。
4. 其他组合按“范围更大优先”处理（保持与上面规则一致，不冲突时取更宽范围）。

说明：

- `CURRENT_DEPT`：按查看人当前主副组织口径。
- `CURRENT_AND_CHILD_DEPT`：按查看人当前主副组织展开口径。
- `CUSTOM_DEPT`：按角色自选组织展开口径（`tb_user_role_dept_expand`）。

### 2.4 最终命中规则

查询箱可见满足任一条件：

1. `B == A`（创建人本人可见）。
2. `A_ALL_DEPTS ∩ B_SCOPE_DEPTS ≠ ∅`（存在至少一个共同组织即命中，例如共同组织为 `2` 则可见）。
3. 若 `B_SCOPE_DEPTS = ALL`，直接命中（查看人可见全部数据）。

## 3. 查询接口口径

- 接口不变：
  - `GET /sys/biz-apply/query/list`
  - `GET /sys/biz-apply/query/page`
  - `GET /sys/biz-apply/query/detail`
- 查询条件不变：`bizApplyId`、`bizDefinitionId`、`title`、`bizStatus`
- 排序不变：`ORDER BY ba.updated_at DESC, ba.id DESC`
- `list/page/detail` 口径必须一致

## 4. 实现蓝图（建议）

### 4.1 Service 侧

1. 计算 `B_SCOPE_DEPTS`（按 2.3 规则）。
2. 计算权限分支标记：
   - `hasAllData=true`：当 `B_SCOPE_DEPTS = ALL`。
   - `hasAllData=false`：其他情况，并将 `B_SCOPE_DEPTS` 作为 SQL 入参。
3. Service 只负责算“查看人的组织权限范围”，不提前把“所有可见创建人ID”整批查出来。
4. 旧做法与新做法说明：
   - 旧做法：先查出大量创建人ID，再拼 `ba.applicant_id IN (id1,id2,id3,...)`。
   - 新做法：不拼大 `IN`，在 SQL 用 `EXISTS` 按每条申请记录判断是否有组织交集。

### 4.2 Mapper 侧

- 保持现有查询字段与排序。
- 权限过滤采用分支 SQL：
  - `hasAllData=true`：跳过权限拼接（不追加组织权限条件）。
  - `hasAllData=false`：追加 `EXISTS` 子查询，按“创建人组织展开集合与 `B_SCOPE_DEPTS` 有交集”过滤。
- `EXISTS` 建议口径：
  - 基于 `tb_user_dept_rel_expand ade`（`ade.user_id = ba.applicant_id`）。
  - `ade.dept_id IN (B_SCOPE_DEPTS)` 即命中。

伪 SQL（用于实现对齐）：

```sql
SELECT ...
FROM tb_biz_apply ba
LEFT JOIN tb_workflow_instance wi ON wi.biz_id = ba.id AND wi.is_deleted = 0
WHERE ba.is_deleted = 0
  -- 其他查询条件：bizApplyId / bizDefinitionId / title / bizStatus
  <if test="qto.hasAllData != true">
    AND (
      ba.applicant_id = #{qto.currentUserId}
      OR EXISTS (
        SELECT 1
        FROM tb_user_dept_rel_expand ade
        WHERE ade.user_id = ba.applicant_id
          AND ade.dept_id IN
          <foreach collection="qto.visibleDeptIdList" item="deptId" open="(" separator="," close=")">
            #{deptId}
          </foreach>
      )
    )
  </if>
ORDER BY ba.updated_at DESC, ba.id DESC
```

分支说明：

- 当 `hasAllData=true` 时，`<if>` 内权限片段完全不拼接，等价于仅保留“基础查询条件”（全部可见）。
- 当 `hasAllData=false` 时，拼接 `EXISTS` 条件，命中逻辑为“本人可见 + 组织交集可见”。

## 5. 验收样例

### 5.1 正向

1. `B` 为创建人：可见。
2. `A` 组织展开为 `{1,2,3}`，`B_SCOPE_DEPTS={2,3}`：可见。
3. `A` 组织展开为 `{1,2,3}`，`B_SCOPE_DEPTS={2,4}`：因共同组织 `2`，可见。
4. `B_SCOPE_DEPTS=ALL`：可见（跳过权限 SQL 拼接）。

### 5.2 反向

1. `A` 组织展开为 `{1,2,3}`，`B_SCOPE_DEPTS={4,5}`：无共同组织，不可见。
2. 详情接口访问不可见记录：返回“记录不存在或无权限查看”。

### 5.3 一致性

1. `list`、`page`、`detail` 命中范围一致。
2. `list` 与 `page` 排序一致。
