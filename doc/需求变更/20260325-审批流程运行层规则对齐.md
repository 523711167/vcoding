# 审批流程运行层规则对齐

## 变更原因

当前工作流运行层已经具备基础表结构与实体，但与最新确认的运行规则仍存在几处偏差：

- `tb_workflow_node_approver_instance` 缺少节点名称、节点类型冗余字段
- `tb_workflow_node_instance` 缺少 `CANCELED` 状态说明
- `tb_workflow_node_approver_instance` 缺少 `WAITING_ADD_SIGN` 状态说明
- `tb_workflow_approval_record.action` 尚未覆盖系统自动路由、并行拆分、并行聚合、自动通过、自动拒绝、提醒等动作口径

为避免后续实现审批引擎时继续散落字符串常量，需先将运行层表结构、实体、枚举与设计文档统一。

## 变更内容

### 1. 运行层枚举补充

新增以下枚举：

- `WorkflowApprovalActionEnum`
- `WorkflowInstanceStatusEnum`
- `WorkflowNodeInstanceStatusEnum`
- `WorkflowNodeApproverInstanceStatusEnum`

用于统一运行层动作与状态编码，避免后续执行引擎继续散写字符串。

### 2. 节点审批人实例表补充节点快照字段

`tb_workflow_node_approver_instance` 新增：

- `node_name`：所属节点名称冗余
- `node_type`：所属节点类型冗余
- `relation_type`：来源关系类型，区分原始审批人与加签审批人
- `source_approver_instance_id`：来源审批人实例 ID，用于记录加签链路

字段取值来源于所属 `tb_workflow_node_instance` 的节点快照，审批人实例创建时一次性写入，后续不回写。
其中：

- `relation_type=ORIGINAL` 表示原始审批人
- `relation_type=ADD_SIGN` 表示加签审批人
- `source_approver_instance_id` 仅在 `ADD_SIGN` 场景下有值，指向发起加签的审批人实例

### 3. 运行层状态口径对齐

`tb_workflow_node_instance.status` 调整为：

- `PENDING`
- `ACTIVE`
- `APPROVED`
- `REJECTED`
- `SKIPPED`
- `CANCELED`
- `TIMEOUT`

说明：

- `SKIPPED` 暂时保留，但当前主流程按“按需创建节点实例”执行，不作为高频状态使用
- `CANCELED` 用于表达节点已激活或运行中，但被撤回、被流程提前终止或被其他结果抢占后强制取消

`tb_workflow_node_approver_instance.status` 调整为：

- `PENDING`
- `WAITING_ADD_SIGN`
- `APPROVED`
- `REJECTED`
- `DELEGATED`
- `CANCELED`

其中 `WAITING_ADD_SIGN` 表示原审批人已发起加签，当前处于等待加签处理完成的中间态。

### 4. 审批记录动作口径对齐

人工动作：

- `SUBMIT`：提交申请
- `APPROVE`：审批通过
- `REJECT`：审批拒绝
- `DELEGATE`：审批转交
- `RECALL`：发起人撤回
- `ADD_SIGN`：发起加签

系统动作：

- `ROUTE`：系统自动路由
- `SPLIT_TRIGGER`：系统触发并行拆分
- `JOIN_ARRIVE`：分支到达并行聚合节点
- `JOIN_PASS`：并行聚合完成并继续流转
- `AUTO_APPROVE`：系统自动审核通过
- `AUTO_REJECT`：系统自动审批拒绝
- `TIMEOUT`：节点超时自动处理触发记录
- `REMIND`：节点超时后发送提醒

超时处理口径：

- 超时策略为 `AUTO_APPROVE` 时，审批记录写 `AUTO_APPROVE`
- 超时策略为 `AUTO_REJECT` 时，审批记录写 `AUTO_REJECT`
- `TIMEOUT` 仅保留为“达到超时处理点”的系统触发动作
- `NOTIFY_ONLY` 不写 `TIMEOUT`，只写 `REMIND`

系统动作统一使用固定系统账号写入：

- `operator_id=0`
- `operator_name=SYSTEM`
