# 流程加签功能开发

## 一、背景

当前系统已支持流程提交、审批、转交、取消等运行能力，但“加签”接口仅保留了入参和路由骨架，尚未具备可执行的运行逻辑，导致审批人在遇到需要临时追加协同审批人的场景时无法落地。

本次需求在现有节点审批人实例模型基础上补齐“前加签”能力，并明确加签链路、状态流转、待办展示和拒绝口径。

## 二、范围

- 新增 `/sys/workflow-biz/add-sign` 运行能力。
- 支持当前审批人在当前活动节点对自己的审批席位发起前加签。
- 加签审批人仍归属当前节点，不新增节点实例。
- 当前版本不扩展为后加签、并签/或签可配置加签策略，也不引入新的数据库字段。

## 三、业务规则

### 3.1 发起条件

- 流程实例必须为 `RUNNING`。
- 节点实例必须属于当前流程实例，且节点状态为 `ACTIVE`。
- 发起人必须是当前审批人实例对应用户本人。
- 当前审批人实例必须满足：
  - `status = PENDING`
  - `is_active = 1`
- 不允许给自己加签。
- 不允许加签给当前节点审批链中已存在的用户。
- 同一次加签请求中的用户不允许重复。
- 加签目标用户必须存在且处于启用状态。

### 3.2 运行口径

- 仅支持“前加签”。
- 当前审批人发起加签后，自己的审批人实例更新为：
  - `status = WAITING_ADD_SIGN`
  - `is_active = 0`
- 新增的加签审批人实例写入当前节点，字段口径如下：
  - `relation_type = ADD_SIGN`
  - `source_approver_instance_id = 发起加签的审批人实例ID`
- 多个加签人按 `addSignUserIds` 顺序顺签。
- 第一位加签人创建后立即激活，其余加签人先置为未激活。

### 3.3 审批结果

- 加签人全部通过后：
  - 当前加签链结束
  - 来源审批人恢复为 `PENDING + is_active = 1`
  - 节点不流转，继续等待来源审批人做最终审批
- 任一加签人拒绝后：
  - 当前节点直接按拒绝处理
  - 其余未完成加签审批人统一取消
  - 当前节点内处于 `WAITING_ADD_SIGN` 的来源审批人统一取消
  - 来源审批人不再继续审批

## 四、展示口径

- 加签审批人作为普通待办进入待办箱。
- 来源审批人在 `WAITING_ADD_SIGN` 期间不出现在待办箱。
- 加签链全部通过且来源审批人恢复后，来源审批人重新进入待办箱。
- 已办箱仍仅展示 `APPROVED`、`REJECTED`、`DELEGATED`，来源审批人在等待加签或被取消时不进入已办箱。

## 五、审计口径

- 发起加签时写入一条 `tb_workflow_approval_record`：
  - `action = ADD_SIGN`
  - `comment` 记录加签说明
  - `extra_data` 记录本次加签用户ID列表
- 加签审批人后续审批仍沿用普通审批记录：
  - 通过写 `APPROVE`
  - 拒绝写 `REJECT`

## 六、实现范围

- `WorkflowBizController`
- `WorkflowLaunchService`
- `WorkflowLaunchServiceImpl`
- `WorkflowNodeApproverInstanceService`
- `WorkflowNodeApproverInstanceServiceImpl`
- `WorkflowApprovalRecordService`
- `WorkflowApprovalRecordServiceImpl`
- 相关运行层单测与发起服务单测

## 七、验收要点

- 发起加签后，来源审批人挂起，加签人进入待办。
- 多个加签人按顺序逐个激活。
- 最后一位加签人通过后，来源审批人恢复待办。
- 任一加签人拒绝后，节点按拒绝结束。
- 给自己加签、给节点内已有审批人加签、重复加签用户、越权加签均返回失败。
