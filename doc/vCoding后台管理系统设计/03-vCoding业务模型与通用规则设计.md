# vCoding 业务模型与通用规则设计

## 1. 设计范围

本文件描述除“审批流程模块设计”和“权限与组织架构设计”之外的通用业务设计内容，主要包括：
- 项目总体说明
- 命名与开发规范
- 技术选型
- 业务定义与业务申请模型
- 字典设计
- 接口通用约束
- 与独立前端工程的协作建议

## 2. 总体说明

项目面向企业内部审批场景，适用于请假、报销、合同等业务。

当前仓库采用前后端分离模式：
- 本仓库仅承载后端 API、认证鉴权、持久化和相关配置。
- HTML 页面、模板文件、静态资源和流程设计器由独立前端工程承载。

## 3. 开发基线规范

### 3.1 命名规范

- 所有数据库表统一使用 `tb_` 前缀。
- 业务表统一使用 `tb_biz_*`。
- 工作流表统一使用 `tb_workflow_*`。
- 用户权限相关表统一使用 `tb_user_*` 或 `tb_sys_*`。
- 实体映射、SQL、初始化脚本和设计文档中的表名必须保持一致。

### 3.2 分层规范

- `Controller` 只负责参数接收、参数校验、调用 `Service`、返回结果。
- 查询接口统一使用 `QTO`。
- 新增与修改接口统一使用 `ETO`。
- 所有查询类必须继承 `BaseQueryQTO`。
- 返回对象统一使用 `VO`。
- 所有接口统一返回 `Resp<T>`。
- 所有分页查询都必须同时提供对应的 `list` 接口。

### 3.3 代码规范

- 所有方法必须补充清晰注释。
- 接口实现方法的注释写在接口定义上。
- 私有方法的注释写在私有方法本身上。
- 所有对外接口、`QTO`、`ETO`、`VO`、统一返回对象都必须补全 Swagger 注解。
- MapStruct 组件统一使用 `*StructMapper` 命名，且更新时只能“传入旧对象，返回新对象”。
- 所有 `Mapper` 都必须显式提供 `removeById` 与 `removeByIds`。

## 4. 技术选型

- 基础框架：`Spring Boot 3`
- JDK：`17`
- ORM：`MyBatis-Plus`
- 数据库：`MySQL 8`

- 对象转换：`MapStruct`

## 5. 业务模型设计

### 5.1 业务申请 `tb_biz_apply`

职责：
- 用一张通用申请表覆盖请假、报销、合同等业务场景。
- 通过 `biz_code` 区分不同业务定义。
- 将差异化字段统一收敛到 `form_data` JSON。
- 提交审批时冗余保存 `workflow_name`，便于历史单据直接展示流程名称。

关键字段：
- `biz_code`
- `title`
- `biz_status`
- `applicant_id`
- `dept_id`
- `form_data`
- `workflow_name`
- `workflow_instance_id`

### 5.2 业务定义 `tb_biz_definition`

职责：
- 定义系统支持的业务定义。
- 通过 `workflow_definition_id` 绑定默认流程定义。
- 通过 `status` 控制是否允许继续发起。

关键规则：
- `biz_code` 全局唯一。
- 业务定义与流程定义的绑定关系仅由 `tb_biz_definition.workflow_definition_id` 维护，`tb_workflow_definition` 不再冗余保存 `biz_code`。
- 删除前需检查历史业务申请和发起权限引用。
- 改绑流程仅影响新提交单据，不影响历史实例。

## 6. 字典设计

核心表：
- `tb_sys_dict_type`
- `tb_sys_dict_item`

设计目标：
- 统一承载表单中的可复用枚举值。
- `form_data` 中枚举字段统一存字典值，不存显示文案。

关键规则：
- `type_code` 全局唯一。
- 同一 `type_code` 下 `item_value` 唯一。
- 字典停用后不再用于新建和编辑，但不影响历史单据展示。

## 7. 接口通用约束

- 统一前缀：`/sys`、`/workflow`
- 查询统一使用 `GET`
- 新增、修改、删除、启停、授权、审批处理统一使用 `POST`
- `POST` 入参统一使用 JSON 请求体
- 查询接口统一返回 `Resp<List<VO>>` 或 `Resp<PageVo<VO>>`
- 详情接口统一返回 `Resp<VO>`

Swagger 地址：
- `/v3/api-docs`
- `/swagger-ui/index.html`

## 8. 与独立前端工程的协作建议

- 后端只提供流程定义、流程实例、权限、业务和字典等 API。
- 流程设计器建议由独立前端工程承载。
- 前端流程图工具优先建议 `LogicFlow`。

## 9. 边界条件与落地建议

- 所有业务异常必须统一转换为标准响应。
- 禁止将异常堆栈、数据库错误或框架内部错误直接返回前端。
- 新增、修改、删除、启停、授权、审批处理等写接口都要考虑幂等性。
- 审批操作需考虑并发控制，防止同一节点被重复处理。
- 条件表达式执行需控制安全边界，避免注入和非法函数调用。
