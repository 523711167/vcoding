# vCoding 权限与组织架构设计

## 1. 设计范围

本文件聚焦系统基础权限域设计，覆盖以下内容：
- 用户、角色、组织、菜单与按钮模型
- 数据权限
- 认证与授权架构
- 当前登录上下文
- 后台权限管理模块边界

## 2. 设计目标

- 建立统一的用户、角色、组织、菜单与权限模型。
- 为后台管理、接口鉴权、数据权限与审批人解析提供稳定基础。
- 保证“认证服务端 + 资源服务端”同进程部署时职责清晰。

## 3. 核心模型

### 3.1 用户

核心表：
- `tb_user`

职责：
- 存储账号、密码、状态与基础资料。
- 作为统一认证登录主体。

约束：
- `username` 全局唯一。
- 停用后不可登录，也不可被新流程解析为审批人。
- 不在用户表中直接固化角色外键和组织外键。

### 3.2 角色

核心表：
- `tb_user_role`
- `tb_user_role_rel`

职责：
- 聚合菜单权限、按钮权限、数据权限和业务发起权限语义。

约束：
- 一个用户可拥有多个角色。
- `code` 全局唯一，且创建后不可修改。
- 停用角色不再参与新的权限加载和审批人解析。

### 3.3 组织

核心表：
- `tb_user_dept`
- `tb_user_dept_rel`
- `tb_user_dept_rel_expand`

职责：
- 维护组织树、主组织、主管关系和层级结构。
- 统一承载集团、公司、部门、岗位四类节点。

#### 组织层级规则

- 组织节点类型固定为：`GROUP`（集团）、`COMPANY`（公司）、`DEPT`（部门）、`POST`（岗位）。
- 顶级根节点必须是 `GROUP`。
- `GROUP` 下只能新增 `COMPANY`。
- `COMPANY` 下只能新增 `DEPT`。
- `DEPT` 下可以新增子 `DEPT` 或 `POST`。
- `POST` 为最小组织单元，岗位节点下不允许再新增任何子组织。
- 组织树不得出现循环引用。

#### 用户与组织关系

- 一个用户可关联多个组织。
- 一个用户有且仅有一个主组织。
- 主组织用于默认组织展示、数据权限口径和主管解析。
- `tb_user_dept_rel` 只记录用户直接选中的组织节点。
- `tb_user_dept_rel_expand` 只记录系统基于直接绑定关系向下展开后的当前生效节点。
- 展开结果保留来源关系，同一目标节点允许因不同来源节点重复出现。

#### 字段设计要求

- `tb_user_dept.org_type`：记录组织节点类型，取值范围为 `GROUP`、`COMPANY`、`DEPT`、`POST`。
- `tb_user_dept.post_type`：记录岗位类型；仅当 `org_type=POST` 时有值。
- `tb_user_dept_rel.org_type`：记录用户绑定时的组织节点类型冗余。
- `tb_user_dept_rel.post_type`：记录用户绑定时的岗位类型冗余；非岗位关联时为空。
- `tb_user_dept_rel_expand.source_rel_id`：记录展开结果来源的直接绑定关系 ID，对应 `tb_user_dept_rel.id`。
- `tb_user_dept_rel_expand.source_dept_id`：记录来源直接绑定的组织节点 ID。
- `tb_user_dept_rel_expand.source_org_type`：记录来源直接绑定节点的组织类型冗余。
- `tb_user_dept_rel_expand.source_post_type`：记录来源直接绑定节点的岗位类型冗余；非岗位来源时为空。
- `tb_user_dept_rel_expand.source_is_primary`：记录该展开关系是否来源于主组织绑定；仅表示来源属性，不代表目标节点本身是主组织。
- `tb_user_dept_rel_expand.dept_id`：记录展开后的目标组织节点 ID。
- `tb_user_dept_rel_expand.org_type`：记录展开后目标节点的组织类型冗余。
- `tb_user_dept_rel_expand.post_type`：记录展开后目标岗位类型冗余；非岗位目标时为空。
- `tb_user_dept_rel_expand.relation_type`：记录展开关系类型，固定为 `SELF` 或 `DESCENDANT`。
- `tb_user_dept_rel_expand.distance`：记录展开距离，自身为 `0`，直接子节点为 `1`。

#### 设计思路说明

- `org_type` 负责表达组织结构语义，用于校验“谁能挂在谁下面”。
- `post_type` 负责表达岗位业务语义，用于区分“同为岗位但职责不同”的节点。
- `org_type` 仅允许在新增组织时确定；编辑组织时不允许修改。
- `tb_user_dept_rel.org_type` 与 `tb_user_dept_rel.post_type` 属于绑定时冗余字段，用于降低后续用户组织查询和岗位解析时的联表复杂度。
- `tb_user_dept_rel` 的绑定语义是“记录用户直接绑定了哪些组织节点”，不是“把该组织整棵子树全部展开写入同一张表”。
- `tb_user_dept_rel_expand` 是派生表，不承载人工授权事实，只承载系统按组织树向下展开后的当前有效节点。
- direct / expand 分表的核心目的是避免“用户直接绑了这个节点”和“因为祖先节点展开继承到了这个节点”两种语义混在同一张表中。
- 展开表保留来源字段，是为了支持后续追溯“这条生效关系来自哪一条直接绑定记录”，并允许同一目标节点因多个来源节点保留多条记录。
- 展开方向固定为向下展开，且每条直接绑定关系必须先生成 1 条 `SELF` 记录，再递归生成全部 `DESCENDANT` 记录。
- 展开表只保存当前有效节点；组织停用、删除、移动、类型调整后，需要按受影响用户重建展开结果。
- 展开维护策略采用“按用户全量重建”，不做单条增量修补；用户组织全量分配时，同一事务内先清空旧展开记录，再按新绑定关系整体重建。
- 若后续需要同步保存当前绑定节点的名称、路径、类型等快照信息，与本设计不冲突。
- 查询“用户直接选中了哪些组织”时只查 `tb_user_dept_rel`；查询“某组织当前覆盖了哪些用户”或“某用户当前生效覆盖了哪些节点”时查 `tb_user_dept_rel_expand`。

### 3.4 菜单与按钮

核心表：
- `tb_sys_menu`
- `tb_user_role_menu`

职责：
- 表达目录、菜单、按钮和接口权限点。

约束：
- 按钮必须挂在菜单下。
- 按钮必须配置全局唯一的 `permission`。
- 隐藏菜单只影响导航展示，不直接改变接口鉴权行为。

## 4. 数据权限设计

核心表：
- `tb_user_role`
- `tb_user_role_dept`

`data_scope` 建议枚举如下：

| 值 | 名称 | 说明 |
|----|------|------|
| `1` | 全部数据 | 查看系统内所有数据 |
| `2` | 自定义部门 | 查看指定部门数据 |
| `3` | 本部门及子部门 | 查看主部门及其子树数据 |
| `4` | 仅本部门 | 查看主部门数据 |
| `5` | 仅本人数据 | 仅查看本人数据 |

规则：
- 用户同时拥有多个角色时，最终数据权限取范围最宽的一档。
- 数据权限主要作用于查询类接口。
- 待办、已办、我的发起等任务视图优先按任务参与关系判断，不应被普通数据权限误拦截。

## 5. 权限协同关系

- 菜单权限控制功能入口是否可见。
- 按钮 `permission` 控制接口操作权限。
- 数据权限控制查询结果范围。
- 业务发起权限控制用户能发起哪些业务。
- 流程规则控制审批单如何流转以及由谁审批。

## 6. 后台管理模块设计

### 6.1 用户管理

核心功能：
- 新增、修改、删除用户
- 用户列表、分页、详情
- 重置密码
- 启停用户
- 用户角色全量分配
- 用户组织全量分配

关键规则：
- 删除前需检查是否存在进行中的审批任务。
- 新增用户时密码统一加密。
- 用户角色与组织变更后需要刷新权限上下文。
- 用户组织全量分配后，需要在同一事务内全量重建该用户的 `tb_user_dept_rel_expand`。

### 6.2 角色管理

核心功能：
- 新增、修改、删除角色
- 角色列表、分页、详情
- 菜单授权
- 数据权限配置
- 查看角色下用户

关键规则：
- 删除前需检查用户关联、业务发起权限引用和流程规则引用。

### 6.3 组织管理

核心功能：
- 新增、修改、移动、删除组织
- 组织树、详情
- 维护组织主管
- 查看组织下用户

关键规则：
- 删除前需检查无子组织、无用户关联、无关键流程引用。
- 移动组织时需重算 `path` 与 `level`。
- 组织新增与移动时必须校验 `org_type` 的父子层级合法性。
- 编辑组织时不允许变更 `org_type`；若需调整组织类型，应通过新建、迁移、删除等方式处理。
- 岗位节点为末级节点，岗位下禁止新增或移动进入任何子组织。
- 集团下只能维护公司，公司下只能维护部门，部门下允许继续维护子部门或岗位。
- 编辑组织时，`post_type` 按当前节点实际 `org_type` 处理：`POST` 节点允许修改，非 `POST` 节点禁止设置。
- 用户绑定组织时，需同步维护 `tb_user_dept_rel.org_type`；若绑定的是岗位节点，还需同步维护 `tb_user_dept_rel.post_type`。
- 用户组织绑定的直接关系写入 `tb_user_dept_rel`，向下展开后的生效关系写入 `tb_user_dept_rel_expand`。
- 组织新增、删除、移动、状态调整、`org_type` 调整、`post_type` 调整后，需要仅重建受影响用户的展开关系。
- 组织名称、编码、排序、主管变更不触发展开关系重建，因为展开表不保存这些快照字段。

### 6.4 菜单管理

核心功能：
- 新增、修改、删除菜单和按钮
- 查询菜单树
- 角色菜单授权

关键规则：
- `permission` 必须保持全局唯一。

## 7. 认证与授权设计

当前项目采用“认证服务端 + 资源服务端”同进程部署模式：

- 认证服务端负责 `/oauth2/token`、`/oauth2/revoke`、`/oauth2/introspect`
- 资源服务端负责 Bearer Token 校验与业务接口保护
- 两者部署在同一个 Jar 中，但通过独立安全过滤链划分职责

### 7.1 Token 策略

- 登录态统一使用 `opaque token`
- 资源服务端基于内省结果判断令牌是否有效
- 该方案用于满足“令牌撤销后立即失效”的需求

### 7.2 用户名密码登录方式

- 统一通过 `/oauth2/token` 扩展自定义 `grant_type=password_login`
- 不新增平行的自定义发 token 登录接口

### 7.3 统一配置项

- `workflow.security.issuer`
- `workflow.security.token-expire-seconds`
- `workflow.security.client-id`
- `workflow.security.client-secret`

## 8. 当前登录上下文

建议上下文包含：
- `userId`
- `roleCodes`
- `primaryDeptId`
- `permissions`
- `dataScope`
- `dataDeptIds`

建议加载顺序：
1. 查询用户角色。
2. 查询角色授权菜单与按钮。
3. 汇总权限标识与菜单树。
4. 计算最终数据权限。
5. 加载主组织。
6. 写入上下文或授权体系可访问位置。

## 9. 主要接口范围

用户相关：
- `/sys/user/*`

角色相关：
- `/sys/role/*`

组织相关：
- `/sys/dept/*`

菜单相关：
- `/sys/menu/*`

认证相关：
- `/oauth2/token`
- `/oauth2/revoke`
- `/oauth2/introspect`
- `/sys/auth/current-user`

## 10. 边界规则

- 数据权限解决的是“能查哪些数据”，不是“是否拥有审批任务”。
- 用户停用、角色停用、组织调整都需要评估对审批解析和数据权限的影响。
- 所有授权类修改建议采用全量替换，并在事务内完成。
- 授权完成后应刷新缓存，或使旧权限上下文失效。
- 主组织口径只认 `tb_user_dept_rel.is_primary`，不得从 `tb_user_dept_rel_expand` 反推主组织。
- `tb_user_dept_rel_expand` 属于派生结果表，不允许手工单条维护，只允许由系统按规则重建。
- 使用展开表查询组织下用户时，结果需按 `user_id` 去重，不能直接把展开表当作唯一用户结果集。

详细建表 SQL、接口清单和原始章节可查看 [vCoding 后台管理系统完整设计归档](./00-vCoding后台管理系统完整设计归档.md)。
