## 一、文档概述

本文档描述一套面向企业内部审批场景的工作流与系统权限设计方案，适用于请假审批、费用报销、合同审批等业务。

本文档目标：
- 统一审批工作流、基础数据、系统权限、后台管理的整体设计口径
- 形成一套可直接指导后续表结构实现、接口设计和后台开发的方案
- 形成统一且可落地的对象命名规范，避免业务域之间出现多套前缀并存

### 1.1 命名最高标准

以下命名规范为本文档最高标准，若与旧命名口径冲突，一律以本标准为准并同步修正文档内容：
- 所有数据库表统一使用 `tb_` 前缀，禁止在同一项目内混用无前缀表名或其他前缀体系
- 业务相关表统一命名为 `tb_biz_*`，如 `tb_biz_apply`、`tb_biz_type`、`tb_biz_type_initiator`
- 用户、角色、组织及其关联表统一命名为 `tb_user*` 体系，如 `tb_user`、`tb_user_role`、`tb_user_dept`、`tb_user_role_rel`、`tb_user_dept_rel`
- 工作流相关表统一命名为 `tb_workflow_*`，如 `tb_workflow_definition`、`tb_workflow_instance`、`tb_workflow_node`
- 系统配置与权限相关表统一命名为 `tb_sys_*` 或 `tb_user_*` 体系，如 `tb_sys_menu`、`tb_sys_dict_type`、`tb_user_role_menu`
- 代码中的实体映射、注解 SQL、初始化 SQL、迁移 SQL、设计文档必须与最终表名保持完全一致

### 1.2 开发最高标准

以下开发规范为本文档最高标准，若与局部接口说明、示例写法或旧开发口径冲突，一律以本标准为准并同步修正文档内容：
- `Controller` 层只负责参数接收、参数校验、调用 `Service` 和返回结果，禁止在 `Controller` 中编写业务逻辑
- `Controller` 层接收参数必须按职责区分对象类型：查询类接口统一使用 `QTO`，所有非查询类接口统一使用 `ETO`
- 查询类接口通过请求参数绑定到 `QTO`，所有非查询类接口通过 JSON 请求体接收 `ETO`
- 所有查询类 `QTO` 必须统一继承 `BaseQueryQTO`，默认复用其中的 `id`、`idList` 查询参数，禁止在各查询对象中重复定义同类基础字段
- `Controller` 层返回对象统一使用 `VO`，不直接返回数据库实体对象、持久化对象或其他内部领域对象
- 所有接口返回结果统一封装为 `Resp<T>`，固定包含 `code`、`msg`、`data` 三个字段，其中 `data` 为泛型业务数据
- 列表查询返回 `Resp<List<VO>>`，分页查询返回 `Resp<PageResult<VO>>`，详情查询返回 `Resp<VO>`
- 所有分页查询都必须同时提供对应的列表查询接口，两者查询条件、查询 SQL、返回字段结构、排序规则、数据权限口径必须保持一致
- 分页查询与列表查询的唯一差异为：分页查询额外携带分页参数，且返回结果额外包含分页信息
- 同一业务模块下，列表接口与分页接口必须复用同一套查询实现，禁止出现字段不一致、条件不一致、排序不一致、数据范围不一致的情况
- 所有删除接口都必须优先支持批量删除，单个删除仅作为批量删除的一种特例处理
- `Service` 层负责承载完整业务逻辑、事务控制和业务规则校验，禁止将事务放在 `Controller` 层或 `Mapper` 层
- `Mapper` 层只负责数据访问，禁止在 SQL 或持久层中编写与职责无关的业务分支逻辑
- 所有 `Mapper` 都必须显式提供 `deleteById` 和 `deleteByIds` 方法，通过注解 SQL 分别执行按主键物理删除和按主键集合批量物理删除；关联表数据删除时，必须先查询关联记录主键，再直接调用对应 `Mapper.deleteById` 或 `Mapper.deleteByIds` 完成删除，禁止继续散落 `Wrapper.delete(...)` 删除写法
- 所有 `StructMapper` 都必须遵循“传入旧对象，返回新对象”的使用规范，禁止通过 `@MappingTarget` 或其他方式直接修改传入对象；对象转换逻辑应尽可能收敛在对应的 `StructMapper` 中实现
- `Entity` 只用于数据库持久化映射，禁止直接作为接口入参或出参
- 所有持久化对象字段都必须添加清晰注释；对于 `id`、`createdAt`、`updatedAt`、`isDeleted` 等公共字段，应优先抽取到统一持久化基类中复用
- 所有方法必须添加清晰注释；若为接口实现方法，应在接口定义的方法上编写注释；若为私有方法，应直接在私有方法上编写注释
- 接口文档统一使用 `Swagger/OpenAPI` 生成，所有对外接口、`QTO`、`VO`、统一返回对象都必须补充完整注解，保证文档可直接用于联调和验收
- 所有 `QTO` 入参必须做统一参数校验，优先使用注解校验，避免重复手写参数判断
- 对于不涉及数据库查询的固定枚举值、状态值、常量范围校验，必须优先在 `Controller` 入参层通过注解完成，禁止继续在 `Service` 层手写同类 `if` 校验
- 所有业务异常必须统一转换为标准响应，禁止将异常堆栈、数据库错误信息或框架内部错误直接返回前端
- 所有新增、修改、删除、启停、授权、审批处理接口都应考虑幂等性，避免重复提交导致数据异常
- 所有表结构应统一审计字段、逻辑删除字段和状态字段口径，保持实现一致性
- 所有枚举类字段统一存值不存显示文案，显示文案通过字典或 `VO` 扩展字段返回
- 所有跨表查询必须关注索引设计、查询性能和数据量边界，禁止默认接受慢 SQL、全表扫描和 `N+1` 查询问题
- 所有权限校验、数据权限校验、业务发起权限校验都应在统一业务入口处理，禁止在多个层级重复分散实现

系统核心能力：
- 可视化流程定义（节点 + 连线 + 条件）
- 多人审批（会签 / 或签 / 顺签）
- 并行处理（并行网关）
- 条件分支（基于表达式）
- 超时催办 / 自动处理
- 业务发起权限控制（角色 / 部门）
- 菜单权限、数据权限、业务发起权限、流程规则统一设计

## 二、技术选型

### 2.1 技术栈组成

- 基础框架：`Spring Boot 3`
- ORM 框架：`MyBatis-Plus`
- 数据库：`MySQL 8.0+`
- 接口文档工具：`Swagger/OpenAPI`

### 2.1.1 数据库连接环境

当前开发环境数据库连接信息如下：

- 主机地址：`localhost`
- 端口：`3306`
- 数据库名：`yuyu`
- 用户名：`user`
- 密码：`user`
- 数据库版本：`MySQL 8`

### 2.2 选型说明

- `Spring Boot 3` 作为项目基础框架，适合承载审批工作流、权限体系、后台管理接口等模块化能力
- `MyBatis-Plus` 用于提升基础 CRUD、分页查询、条件构造、通用字段维护的开发效率，适合本方案中大量主数据表、配置表、实例表的实现
- `MySQL 8.0+` 用于支撑流程定义、流程实例、组织树、权限配置、JSON 表单数据等存储需求，能够满足索引、事务、一致性和 JSON 字段的设计要求
- 当前仓库采用前后端分离模式，前端页面与流程设计器由独立前端工程承载；本仓库仅负责 API、认证、权限、流程数据和持久化能力

### 2.3 流程图工具选型建议

本方案更推荐由独立前端工程使用 `LogicFlow` 绘制审批流程图。

推荐原因：
- 当前工作流设计采用的是自定义审批流模型，而不是完整 BPMN 标准模型
- 后端核心结构是 `tb_workflow_node + tb_workflow_transition + tb_workflow_node_approver`，与 `LogicFlow` 的节点和边模型更容易直接映射
- 在审批流场景下，常见需求是“节点配置 + 连线条件 + 属性面板 + 拖拽排版”，`LogicFlow` 更贴近这类场景的实现方式
- 相比标准 BPMN 方案，`LogicFlow` 的前后端映射成本更低，更适合当前项目快速落地

其他可选方案说明：
- `AntV X6`：适合做高度定制的通用图编辑器，自由度高，但需要自行封装更多流程语义，整体开发成本更高
- `bpmn-js`：适合严格遵循 BPMN 规范的平台型流程系统，但当前设计并非 BPMN 原生建模，若强行采用，前端 BPMN 元素与后端自定义表结构之间会增加额外映射复杂度

因此，本项目建议优先采用：`独立前端工程 + LogicFlow + 后端 API` 的集成模式。

### 2.4 前后端分离集成建议

#### 2.4.1 页面职责划分

建议采用“独立前端工程承载后台管理页面与流程画布，后端仓库仅提供 API”的方式。

可拆分为以下页面：
- 流程定义列表页：展示流程编码、名称、版本、状态、绑定业务类型
- 流程设计页：用于绘制节点、连线、配置审批人、配置条件表达式
- 流程详情页：查看已发布版本的结构和配置
- 流程实例页：查看某次审批实例的流转轨迹、当前节点和审批记录

其中：
- 独立前端工程负责页面骨架、表单区域、属性面板区域、按钮权限控制、基础数据展示
- `LogicFlow` 负责流程图画布、节点拖拽、节点连线、选中态、画布缩放与节点结构展示

#### 2.4.2 页面结构建议

流程设计页建议包含以下区域：
- 左侧：节点物料区，如开始节点、审批节点、条件节点、并行网关、结束节点
- 中间：`LogicFlow` 流程画布区
- 右侧：属性配置面板，用于编辑节点名称、审批人类型、审批模式、超时策略、连线条件表达式等配置
- 顶部：保存草稿、发布、复制版本、返回列表等操作按钮

#### 2.4.3 前后端交互建议

建议采用“加载定义时组装前端流程 JSON，保存时再拆回后端表结构”的方式。

后端对前端返回的流程设计数据可统一抽象为：
- 流程基础信息：流程编码、名称、业务类型、版本、状态
- 节点集合：节点标识、节点类型、名称、坐标、节点配置
- 连线集合：起点、终点、条件表达式、优先级、标签
- 审批人配置集合：节点审批人类型、审批人值、顺序等

#### 2.4.4 与表结构的映射关系

建议映射方式如下：
- 前端 `definition` 对应 `tb_workflow_definition`
- 前端 `nodes` 对应 `tb_workflow_node`
- 前端 `transitions` 对应 `tb_workflow_transition`
- 前端 `approvers` 对应 `tb_workflow_node_approver`

这样做的好处是：
- 页面模型清晰
- 保存时容易落库
- 加载时容易回显
- 后续扩展节点配置时，只需扩展节点 `config` 或属性面板即可

#### 2.4.5 为什么推荐这种组合

采用“独立前端工程 + LogicFlow + 后端 API”的组合，适合当前项目的原因主要有：
- 当前项目采用前后端分离架构，前端页面不应继续内嵌在后端仓库中
- 流程设计器只是前端能力的一部分，前端可独立负责页面交互、路由和组件体系
- `LogicFlow` 可以专注解决流程图绘制与交互，而 `Spring Boot 3` 继续负责业务接口与页面组织
- 对当前项目而言，这种职责边界更清晰，交付和维护都更稳妥

### 2.5 落地约束建议

- 所有表统一使用 `InnoDB` 存储引擎，确保事务一致性与行级锁能力
- 时间字段统一使用 `DATETIME`，由应用层和数据库默认值共同维护
- 主键统一使用 `BIGINT`
- 流程表单数据、节点扩展配置等非固定结构字段可使用 `JSON` 类型，前提是数据库版本为 `MySQL 8.0+`
- 基于 `MyBatis-Plus` 可统一处理逻辑删除、审计字段自动填充、分页查询和基础 Service 封装
- 基于 `Spring Boot 3` 建议按“控制层 / 应用服务层 / 领域逻辑层 / 持久层”进行模块拆分，避免流程编排逻辑直接散落在 Controller 或 Mapper 中
- 前端独立工程可负责流程列表、版本管理、设计器入口、审批详情等管理页面，流程图本身由前端嵌入 `LogicFlow` 画布实现；当前后端仓库不再承载 HTML 页面或模板文件

## 三、总体设计原则

### 2.1 设计原则

- 用户与角色必须通过 `tb_user_role_rel` 维护，不在 `tb_user` 上直接固化 `role_id`
- 用户与组织必须通过 `tb_user_dept_rel` 维护，不在 `tb_user` 上直接固化 `dept_id`
- 一个用户可关联多个角色、多个组织，但主组织有且仅有一个
- 菜单权限、数据权限、业务发起权限、流程审批规则分别建模，但统一以用户、角色、组织三类主体进行解析
- 流程与业务类型绑定，用户拥有的是业务发起权限，而不是对某个流程的单独发起权限
- 流程实例创建后绑定具体 `definition_id`，不随流程版本升级受影响
- 当前阶段沿用 `tb_sys_menu + tb_user_role_menu + permission` 方案，保持与现有设计一致
- 后续如需更细粒度接口级权限，可增量扩展 `tb_sys_permission`、`tb_user_role_permission`

### 2.2 总体架构

本系统整体采用“审批工作流引擎 + 系统基础数据 + 权限体系 + 后台管理”的设计。

从职责上可分为四层：

| 层级 | 说明 | 核心表 |
|------|------|--------|
| 工作流定义层 | 描述流程模板、节点、连线、审批人配置 | `tb_workflow_definition`、`tb_workflow_node`、`tb_workflow_transition`、`tb_workflow_node_approver` |
| 工作流运行层 | 描述流程实例、节点实例、审批人实例、审批记录 | `tb_workflow_instance`、`tb_workflow_node_instance`、`tb_workflow_node_approver_instance`、`tb_workflow_approval_record` |
| 系统基础数据层 | 描述用户、角色、组织、菜单、业务申请、业务发起权限等基础模型 | `tb_user`、`tb_user_role`、`tb_user_dept`、`tb_sys_menu`、`tb_biz_apply`、`tb_biz_type`、`tb_biz_type_initiator` |
| 授权与管理层 | 描述角色授权、数据权限、后台功能、接口边界 | `tb_user_role_rel`、`tb_user_dept_rel`、`tb_user_role_menu`、`tb_user_role_dept` |

### 2.3 权限总体架构

系统统一采用“RBAC + 组织树 + 数据权限 + 业务发起权限 + 流程审批规则”的权限架构，避免形成独立于审批体系之外的第二套模型。

| 层级 | 控制对象 | 核心表 | 说明 |
|------|----------|--------|------|
| 认证层 | 用户身份 | `tb_user` | 负责登录认证、账号状态、基础资料 |
| 角色层 | 权限聚合单元 | `tb_user_role`、`tb_user_role_rel` | 角色是菜单权限、数据权限、业务发起权限的重要承载体 |
| 组织层 | 树形归属关系 | `tb_user_dept`、`tb_user_dept_rel` | 组织用于树形管理、主管解析、数据范围口径 |
| 功能权限层 | 菜单/按钮权限 | `tb_sys_menu`、`tb_user_role_menu` | 控制页面可见范围和按钮操作权限 |
| 数据权限层 | 数据可见范围 | `tb_user_role.data_scope`、`tb_user_role_dept` | 控制业务数据可查询范围 |
| 业务发起权限层 | 能否发起某类业务 | `tb_biz_type_initiator` | 与业务类型联动，控制谁可发起某类业务 |
| 流程规则层 | 节点审批规则 | `tb_workflow_node_approver` | 与审批工作流联动，控制节点最终解析给谁审批 |

## 三、核心概念

| 概念 | 说明 |
|------|------|
| 业务类型（Biz） | 系统支持的审批业务种类，如请假、报销、合同 |
| 业务发起权限 | 用户是否可以发起某类业务，由角色或组织决定 |
| 流程定义（Definition） | 可复用的流程模板，包含节点、连线和审批规则 |
| 流程实例（Instance） | 某次具体审批的运行记录 |
| 节点（Node） | 流程中的处理单元 |
| 流转（Transition） | 节点间的有向边，可附带条件表达式 |
| 审批人（Approver） | 在某节点处理审批任务的人 |
| 角色（Role） | 菜单权限、数据权限、业务发起权限和审批语义的重要聚合单元 |
| 组织（Dept） | 树形组织结构，同时参与主管解析和数据权限计算 |
| 主组织（Primary Dept） | 用户默认归属组织，用于主管解析、数据权限口径、默认归属部门 |

### 3.1 节点类型

| 类型值 | 名称 | 说明 |
|--------|------|------|
| `START` | 开始节点 | 流程的唯一入口 |
| `APPROVAL` | 审批节点 | 需要审批人操作的节点 |
| `CONDITION` | 条件节点 | 根据条件表达式决定走向 |
| `PARALLEL_SPLIT` | 并行拆分网关 | 同时激活多条分支 |
| `PARALLEL_JOIN` | 并行聚合网关 | 等待所有分支完成后继续 |
| `END` | 结束节点 | 流程的终止节点 |

### 3.2 审批人类型

| 值 | 说明 |
|----|------|
| `USER` | 直接指定具体用户 ID |
| `ROLE` | 指定角色，运行时解析该角色下所有有效用户 |
| `DEPT` | 指定部门，运行时解析该部门下候选审批人 |
| `INITIATOR_DEPT_LEADER` | 动态取发起人主组织的主管 |

## 四、业务、流程与权限的关系

### 4.1 关系总览

```text
tb_user ──────────────── tb_user_role_rel ──────────── tb_user_role
    │                                                       │
    │                                                       ├── tb_user_role_menu ──── tb_sys_menu (type=1/2/3)
    │                                                       │
    │                                                       └── tb_user_role_dept ──── tb_user_dept
    │
    └── tb_user_dept_rel ──────────────────────────────── tb_user_dept

tb_biz_type ──────────────── tb_biz_type_initiator
   │
   ├── tb_workflow_definition.id（via workflow_definition_id）
   └── tb_biz_apply.biz_code
            │
            ├── form_data（业务申请数据快照）
            └── tb_workflow_instance （via workflow_instance_id）

tb_sys_dict_type ──────────────── tb_sys_dict_item
```

### 4.2 关键关系说明

- `tb_biz_type` 定义“系统有哪些业务类型”，核心编码统一为 `biz_code`
- `tb_biz_type` 维护业务名称、状态、说明以及默认绑定的流程定义 ID，不再单独维护业务字段配置明细
- `tb_biz_type_initiator` 定义“哪些角色可以发起该业务类型”
- `tb_biz_type.workflow_definition_id` 指向当前业务绑定的流程定义版本，业务发起时以该字段作为流程选择依据
- `tb_workflow_definition.form_key` 可作为流程侧记录业务编码的辅助字段，便于流程定义查询与展示
- `tb_biz_apply.biz_code` 标识“这张业务单据属于哪个业务类型”
- `tb_biz_apply.form_data` 保存业务申请提交时的实际数据快照，字段结构由前后端按业务类型约定
- 单据提交时，先根据 `tb_biz_apply.biz_code` 查询 `tb_biz_type` 获取 `workflow_definition_id`，再创建 `tb_workflow_instance`

### 4.3 权限模型总结

| 权限维度 | 控制载体 | 实现方式 |
|----------|----------|----------|
| 菜单访问权限 | 角色（`tb_user_role`） | `tb_user_role_menu` → `tb_sys_menu(type=1/2)` |
| 按钮操作权限 | 角色（`tb_user_role`） | `tb_user_role_menu` → `tb_sys_menu(type=3).permission` |
| 数据权限 | 角色（`tb_user_role`） | `tb_user_role.data_scope` + `tb_user_role_dept` |
| 业务发起权限 | 角色（`tb_user_role`） | `tb_biz_type_initiator` |
| 审批流规则 | 角色 / 组织 | `tb_workflow_node_approver.approver_type` |

### 4.4 对象职责边界

| 对象 | 核心职责 | 不承担的职责 |
|------|----------|--------------|
| `tb_user` | 存储账号信息、登录凭证、启停状态 | 不直接存储角色外键、组织外键 |
| `tb_user_role` | 聚合功能权限、数据权限、业务发起权限语义 | 不直接表示用户主组织 |
| `tb_user_dept` | 表达组织树、部门主管、层级结构 | 不直接表达用户功能权限 |
| `tb_sys_menu` | 表达前端菜单树和按钮权限点 | 不直接承载数据权限规则 |
| `tb_user_role_rel` | 维护用户与角色的多对多关系 | 不存业务字段 |
| `tb_user_dept_rel` | 维护用户与组织的多对多关系及主组织 | 不存菜单权限 |
| `tb_user_role_menu` | 维护角色与菜单/按钮的授权关系 | 不存用户直授权信息 |
| `tb_biz_type_initiator` | 维护业务类型与发起主体的授权关系 | 不负责审批节点解析 |
| `tb_workflow_definition` | 描述流程模板及版本 | 不负责用户发起权限判断 |

## 五、系统基础数据设计

> 本章描述支撑审批工作流运行的系统基础数据表，包括业务申请、组织、角色、用户、菜单及其关联关系。

### 5.1 业务申请表 `tb_biz_apply`

用一张通用表覆盖所有业务申请场景（请假、报销、合同等），通过 `biz_code` 字段区分业务类型，差异化字段存入 `form_data` JSON 列。

```sql
CREATE TABLE `tb_biz_apply` (
  `id`                   BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `biz_code`             VARCHAR(64)  NOT NULL               COMMENT '业务编码：LEAVE=请假 EXPENSE=报销 CONTRACT=合同 等',
  `title`                VARCHAR(200) NOT NULL               COMMENT '申请标题',
  `status`               VARCHAR(16)  NOT NULL DEFAULT 'DRAFT' COMMENT '申请状态：DRAFT=草稿 PENDING=审批中 APPROVED=已通过 REJECTED=已拒绝 CANCELED=已撤回',
  `applicant_id`         BIGINT       NOT NULL               COMMENT '申请人用户ID',
  `applicant_name`       VARCHAR(64)  NOT NULL               COMMENT '申请人姓名（冗余）',
  `dept_id`              BIGINT                              COMMENT '申请人所属部门ID（冗余）',
  `form_data`            JSON                                COMMENT '业务差异化字段（按biz_code存储申请数据快照）',
  `workflow_instance_id` BIGINT                              COMMENT '关联的审批工作流实例ID（tb_workflow_instance.id），提交后回写',
  `submitted_at`         DATETIME                            COMMENT '提交审批时间',
  `finished_at`          DATETIME                            COMMENT '审批完成时间',
  `created_at`           DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at`           DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted`           TINYINT      NOT NULL DEFAULT 0     COMMENT '软删除标记：0=正常 1=已删除',
  PRIMARY KEY (`id`),
  KEY `idx_applicant_id` (`applicant_id`),
  KEY `idx_biz_code_status` (`biz_code`, `status`),
  KEY `idx_workflow_instance_id` (`workflow_instance_id`)
) ENGINE=InnoDB COMMENT='通用业务申请表';
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `biz_code` | `VARCHAR(64)` | 业务类型编码，对应 `tb_biz_type.biz_code` |
| `status` | `VARCHAR(16)` | 申请单自身状态，随工作流实例状态同步更新 |
| `form_data` | `JSON` | 按 `biz_code` 存储业务申请提交的数据快照；其中枚举类字段统一存字典项值，不存显示文案 |
| `workflow_instance_id` | `BIGINT` | 提交审批后关联到 `tb_workflow_instance.id`，草稿阶段为 `NULL` |

`form_data` 示例（请假申请）：

```json
{
  "leave_type": "ANNUAL",
  "start_date": "2026-03-10",
  "end_date": "2026-03-12",
  "leave_days": 3,
  "reason": "年假休息"
}
```

`form_data` 示例（费用报销）：

```json
{
  "expense_type": "TRAVEL",
  "amount": 2580.00,
  "currency": "CNY",
  "items": [
    { "name": "机票", "amount": 1800.00 },
    { "name": "住宿", "amount": 780.00 }
  ]
}
```

`form_data` 存储口径说明：
- 表单中的枚举字段统一存字典项值，如 `ANNUAL`、`TRAVEL`、`CNY`
- `form_data` 只保存业务提交时的值，不保存字典显示文案，如“年假”“差旅”“人民币”
- 页面展示或导出时，再根据 `type_code + item_value` 解析出对应 `item_label`
- 已入库业务单据按提交时保存的值参与流程流转，不因后续字典文案调整而回写历史单据

关联关系说明：
- `tb_biz_apply.workflow_instance_id` → `tb_workflow_instance.id`
- 申请提交时创建工作流实例，将实例 ID 回写至 `tb_biz_apply.workflow_instance_id`
- 工作流实例状态变更（通过/拒绝/撤回）时，同步更新 `tb_biz_apply.status`

### 5.2 业务类型表 `tb_biz_type`

记录系统支持的业务类型（如请假、报销、合同等），并直接维护该业务默认绑定的流程定义 ID。

```sql
CREATE TABLE `tb_biz_type` (
  `id`                     BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `biz_code`               VARCHAR(64)  NOT NULL               COMMENT '业务编码（全局唯一），如 LEAVE、EXPENSE、CONTRACT',
  `biz_name`               VARCHAR(128) NOT NULL               COMMENT '业务名称，如 请假申请、费用报销',
  `biz_desc`               VARCHAR(500)                        COMMENT '业务说明，用于描述适用场景、填写要求等',
  `workflow_definition_id` BIGINT       NOT NULL               COMMENT '绑定的流程定义ID（tb_workflow_definition.id）',
  `status`                 TINYINT      NOT NULL DEFAULT 1     COMMENT '状态：1=正常 0=停用',
  `created_by`             BIGINT       NOT NULL               COMMENT '创建人用户ID（关联tb_user.id）',
  `created_at`             DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at`             DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted`             TINYINT      NOT NULL DEFAULT 0     COMMENT '软删除标记：0=正常 1=已删除',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_biz_code` (`biz_code`),
  KEY `idx_workflow_definition_id` (`workflow_definition_id`)
) ENGINE=InnoDB COMMENT='业务类型表';
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `biz_code` | `VARCHAR(64)` | 业务编码，全局唯一，建议全大写英文下划线风格 |
| `biz_name` | `VARCHAR(128)` | 业务名称，用于页面展示和单据分类 |
| `biz_desc` | `VARCHAR(500)` | 业务说明，用于描述该业务的适用范围和填写要求 |
| `workflow_definition_id` | `BIGINT` | 当前业务默认绑定的流程定义 ID，创建申请时据此确定流程 |
| `status` | `TINYINT` | 状态：1=正常 0=停用；停用后不可再发起新申请 |

说明：
- `tb_biz_type` 表示审批系统支持的业务类型，是业务申请入口、流程绑定和发起权限控制的业务主维度
- 基础版只维护业务名称、业务说明、默认流程定义和启停状态，不再单独维护业务字段配置表
- 业务申请提交时，先根据 `biz_code` 查询 `tb_biz_type`，读取 `workflow_definition_id`，再创建对应的 `tb_workflow_instance`
- `tb_biz_apply.form_data` 作为业务申请数据快照保存，字段内容由具体业务场景自行约定
- 字典表表示可复用的枚举维度，如请假类型、费用类型、币种等，用于承载业务申请中的通用枚举值
- `tb_biz_type` 与字典表职责不同，前者用于定义“是什么业务”，后者用于定义“业务中可复用的枚举项”，两者不能混用

### 5.3 字典类型表 `tb_sys_dict_type`

用于定义系统内可复用的字典分类，如请假类型、费用类型、币种等，供业务申请中的枚举字段统一引用。

```sql
CREATE TABLE `tb_sys_dict_type` (
  `id`          BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',

  `type_name`   VARCHAR(128) NOT NULL               COMMENT '字典类型名称，如 请假类型、费用类型、币种',
  `status`      TINYINT      NOT NULL DEFAULT 1     COMMENT '状态：1=正常 0=停用',
  `sort_order`  INT          NOT NULL DEFAULT 0     COMMENT '排序值，升序展示',
  `remark`      VARCHAR(255)                        COMMENT '备注说明',
  `created_at`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted`  TINYINT      NOT NULL DEFAULT 0     COMMENT '软删除标记：0=正常 1=已删除',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_type_code` (`type_code`)
) ENGINE=InnoDB COMMENT='字典类型表';
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `type_code` | `VARCHAR(64)` | 字典类型编码，全局唯一，推荐全大写下划线风格 |
| `status` | `TINYINT` | 停用后不再用于新的表单选项加载，但不影响历史数据展示 |
| `sort_order` | `INT` | 字典类型列表展示排序 |
| `remark` | `VARCHAR(255)` | 对该字典类型用途的补充说明 |

使用说明：
- 一个 `type_code` 代表一个可复用枚举维度
- 字典类型本身不直接参与审批流转，仅为业务申请中的枚举字段提供候选值定义
- 建议由后台统一维护，如 `LEAVE_TYPE`、`EXPENSE_TYPE`、`CURRENCY`

与业务申请衔接示例：
- 请假单字段 `leave_type` 关联 `type_code='LEAVE_TYPE'`
- 报销单字段 `expense_type` 关联 `type_code='EXPENSE_TYPE'`
- 报销单字段 `currency` 关联 `type_code='CURRENCY'`

### 5.4 字典项表 `tb_sys_dict_item`

用于维护某个字典类型下的具体可选项，是业务申请中枚举字段的实际取值来源。

```sql
CREATE TABLE `tb_sys_dict_item` (
  `id`          BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `type_code`   VARCHAR(64)  NOT NULL               COMMENT '字典类型编码（关联 tb_sys_dict_type.type_code）',
  `item_value`  VARCHAR(64)  NOT NULL               COMMENT '字典项值，如 ANNUAL、TRAVEL、CNY',
  `item_label`  VARCHAR(128) NOT NULL               COMMENT '字典项显示名称，如 年假、差旅、人民币',
  `status`      TINYINT      NOT NULL DEFAULT 1     COMMENT '状态：1=正常 0=停用',
  `sort_order`  INT          NOT NULL DEFAULT 0     COMMENT '排序值，升序展示',
  `remark`      VARCHAR(255)                        COMMENT '备注说明',
  `created_at`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted`  TINYINT      NOT NULL DEFAULT 0     COMMENT '软删除标记：0=正常 1=已删除',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_type_value` (`type_code`, `item_value`),
  KEY `idx_type_code` (`type_code`)
) ENGINE=InnoDB COMMENT='字典项表';
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `type_code` | `VARCHAR(64)` | 标识该字典项属于哪个字典类型 |
| `item_value` | `VARCHAR(64)` | 业务存储值，推荐全大写英文或编码值 |
| `item_label` | `VARCHAR(128)` | 页面展示文案 |
| `status` | `TINYINT` | 停用后不再允许在新单据中选择 |
| `sort_order` | `INT` | 同一字典类型下的展示排序 |

使用说明：
- 同一 `type_code` 下，`item_value` 必须唯一
- 业务申请提交时只保存 `item_value`，展示时再转换为 `item_label`
- 字典项支持按状态启停，便于平滑下线旧选项

与业务申请衔接示例：
- `type_code='LEAVE_TYPE', item_value='ANNUAL', item_label='年假'`
- `type_code='EXPENSE_TYPE', item_value='TRAVEL', item_label='差旅'`
- `type_code='CURRENCY', item_value='CNY', item_label='人民币'`

示例：若请假单 `form_data` 中保存 `"leave_type": "ANNUAL"`，则页面展示时根据 `LEAVE_TYPE + ANNUAL` 解析为“年假”。

### 5.5 业务发起权限表 `tb_biz_type_initiator`

控制哪些角色可以发起指定业务类型。流程本身不单独配置发起权限；用户只要拥有某业务类型对应角色的权限，即可发起该业务绑定的已发布流程。

设计约束：基础版仅按角色控制，一条记录只表示一个业务类型对应一个角色。

```sql
CREATE TABLE `tb_biz_type_initiator` (
  `id`              BIGINT      NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `biz_code`        VARCHAR(64) NOT NULL               COMMENT '业务编码（关联 tb_biz_type.biz_code）',
  `initiator_type`  VARCHAR(16) NOT NULL DEFAULT 'ROLE' COMMENT '发起主体类型：固定为 ROLE',
  `initiator_value` BIGINT      NOT NULL               COMMENT '发起主体值：角色ID（关联 tb_user_role.id）',
  `created_at`      DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `is_deleted`      TINYINT     NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_biz_initiator` (`biz_code`, `initiator_type`, `initiator_value`),
  KEY `idx_biz_code` (`biz_code`)
) ENGINE=InnoDB COMMENT='业务发起权限表';
```

使用说明：
- 一条记录只对应一个业务类型和一个角色
- 若一个业务允许多个角色发起，则插入多条记录
- 发起校验时，先按业务类型判断当前用户角色是否命中该业务的发起权限；命中后，再选择该业务绑定的已发布流程发起
- 基础版不按部门单独控制发起权限，统一收敛到角色授权

示例：
- `biz_code='LEAVE', initiator_type='ROLE', initiator_value=3`：角色 ID 为 3 的用户可发起请假业务
- `biz_code='LEAVE', initiator_type='ROLE', initiator_value=4`：角色 ID 为 4 的用户可发起请假业务
- `biz_code='EXPENSE', initiator_type='ROLE', initiator_value=6`：角色 ID 为 6 的用户可发起报销业务

### 5.6 部门表 `tb_user_dept`

支持树形结构，通过 `parent_id`、`path`、`level` 三个字段实现高效树形查询，避免递归操作。

```sql
CREATE TABLE `tb_user_dept` (
  `id`          BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `parent_id`   BIGINT       NOT NULL DEFAULT 0     COMMENT '父部门ID，顶级部门为0',
  `name`        VARCHAR(100) NOT NULL               COMMENT '部门名称',
  `code`        VARCHAR(64)                         COMMENT '部门编码（同级唯一）',
  `path`        VARCHAR(500) NOT NULL               COMMENT '祖级路径，格式：/1/5/12/，含自身ID，支持快速子树查询',
  `level`       INT          NOT NULL DEFAULT 1     COMMENT '层级深度，顶级为1，逐层递增',
  `sort_order`  INT          NOT NULL DEFAULT 0     COMMENT '同级排序值，升序展示',
  `leader_id`   BIGINT                              COMMENT '部门主管用户ID（关联tb_user.id），用于审批流动态解析直属上级',
  `leader_name` VARCHAR(64)                         COMMENT '部门主管姓名（冗余）',
  `status`      TINYINT      NOT NULL DEFAULT 1     COMMENT '状态：1=正常 0=停用',
  `created_at`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted`  TINYINT      NOT NULL DEFAULT 0     COMMENT '软删除标记：0=正常 1=已删除',
  PRIMARY KEY (`id`),
  KEY `idx_parent_id` (`parent_id`),
  KEY `idx_path` (`path`(255))
) ENGINE=InnoDB COMMENT='部门表';
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `parent_id` | `BIGINT` | `0` 表示顶级部门（根节点） |
| `path` | `VARCHAR(500)` | 如总公司(1)→技术部(5)→后端组(12)，则 `path='/1/5/12/'` |

`path` 字段的树形查询优势：

```sql

SELECT * FROM tb_user_dept WHERE id IN (1, 5) AND is_deleted = 0;
```

新增子部门时的 path 计算规则：
- 顶级部门：`path = '/{新部门ID}/'`
- 非顶级部门：`path = parent.path + '{新部门ID}/'`

### 5.10 用户-部门关联表 `tb_user_dept_rel`

一个用户可属于多个部门，但有且仅有一个主部门（`is_primary=1`），工作流解析“发起人部门主管”时以主部门为准。

```sql
CREATE TABLE `tb_user_dept_rel` (
  `id`         BIGINT   NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `user_id`    BIGINT   NOT NULL               COMMENT '用户ID（关联tb_user.id）',
  `dept_id`    BIGINT   NOT NULL               COMMENT '部门ID（关联tb_user_dept.id）',
  `is_primary` TINYINT  NOT NULL DEFAULT 0     COMMENT '是否主部门：1=是 0=否',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_tb_user_dept` (`user_id`, `dept_id`),
  KEY `idx_dept_id` (`dept_id`)
) ENGINE=InnoDB COMMENT='用户-部门关联表';
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `is_primary` | `TINYINT` | 每个用户只能有一个主部门，业务层保证唯一性 |

主部门唯一性约束：
- 新增/修改关联时，若 `is_primary=1`，先将该用户其他所有记录的 `is_primary` 置为 0，再写入新记录
- 若用户无主部门，系统自动将第一条关联记录设为主部门
- 推荐通过事务内应用层逻辑或触发器保证唯一性

为什么要求“有且仅有一个主部门”：
- 解析 `INITIATOR_DEPT_LEADER` 时，需要明确按哪个部门的 `leader_id` 查找主管
- 数据权限为“本部门”或“本部门及子部门”时，需要明确以哪个部门作为范围起点
- 业务单据创建时，如需默认带出归属部门，需要统一默认值
- 用户列表、个人信息、统计报表等场景，需要一个稳定的默认归属部门用于展示和归口分析

### 5.11 菜单/按钮表 `tb_sys_menu`

菜单和按钮统一存放在一张表 `tb_sys_menu`，通过 `type` 字段区分类型，通过 `parent_id` 维护层级关系。

```sql

  `path`        VARCHAR(256)                        COMMENT '前端路由地址，目录/菜单类型有效，按钮为空',

  `icon`        VARCHAR(64)                         COMMENT '图标标识，目录/菜单类型有效',
  `sort_order`  INT          NOT NULL DEFAULT 0     COMMENT '同级排序值，升序展示',
  `visible`     TINYINT      NOT NULL DEFAULT 1     COMMENT '是否显示在导航菜单中：1=显示 0=隐藏（按钮不参与此逻辑）',
  `status`      TINYINT      NOT NULL DEFAULT 1     COMMENT '状态：1=正常 0=停用',
  `created_at`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at`  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted`  TINYINT      NOT NULL DEFAULT 0     COMMENT '软删除标记：0=正常 1=已删除',

```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | `TINYINT` | `1`=目录；`2`=菜单；`3`=按钮 |
| `permission` | `VARCHAR(128)` | 后端接口鉴权使用。目录可为空，按钮必填 |

| `component` | `VARCHAR(256)` | 前端组件路径，目录/按钮类型为空 |

- 按钮只能挂在菜单节点下

## 六、权限设计

### 6.1 关系模型与约束规则

#### 用户与角色
- 一个用户可拥有多个角色
- 一个角色可分配给多个用户
- 通过 `tb_user_role_rel` 建模
- 授权方式采用全量替换，便于后台管理和权限缓存刷新

#### 用户与组织
- 一个用户可属于多个组织
- 一个用户有且仅有一个主组织
- 通过 `tb_user_dept_rel` 建模，使用 `is_primary` 标识主组织
- 主组织用于默认组织展示、数据权限口径、主管解析、单据默认归属部门

#### 角色与菜单
- 一个角色可关联多个菜单/按钮
- 一个菜单/按钮可授权给多个角色
- 通过 `tb_user_role_menu` 建模
- 当前阶段不做用户直授权

#### 角色与数据范围
- 一个角色对应一个 `data_scope`
- 当 `data_scope=2` 时，通过 `tb_user_role_dept` 绑定多个部门
- 用户最终数据权限取全部角色中范围最宽的一档

#### 业务类型与发起权限
- 一个业务类型可配置多个发起主体
- 发起主体可以是角色或部门
- 通过 `tb_biz_type_initiator` 建模
- 用户命中任一发起主体，即可发起该业务

### 6.2 数据权限设计

部门权限控制用户能看到哪些业务数据（如审批单列表、用户列表等）。

`data_scope` 枚举说明：

| 值 | 名称 | 说明 |
|----|------|------|
| `1` | 全部数据 | 可查看系统内所有数据，通常分配给超级管理员 |
| `2` | 自定义部门 | 可查看 `tb_user_role_dept` 中配置的指定部门数据 |
| `3` | 本部门及子部门 | 可查看用户主部门及其所有子孙部门的数据 |
| `4` | 仅本部门 | 只能查看用户主部门的数据 |
| `5` | 仅本人数据 | 只能查看自己创建/发起的数据 |

数据权限过滤 SQL 模板：

```sql
-- data_scope=2：自定义部门
AND tb_biz_apply.dept_id IN (
    SELECT dept_id FROM tb_user_role_dept WHERE role_id = #{roleId}
)

-- data_scope=3：本部门及子部门
AND tb_biz_apply.dept_id IN (
    SELECT id FROM tb_user_dept
    WHERE path LIKE CONCAT(
        (SELECT path FROM tb_user_dept WHERE id = #{currentUserPrimaryDeptId}), '%'
    ) AND is_deleted = 0
)

-- data_scope=4：仅本部门
AND tb_biz_apply.dept_id = #{currentUserPrimaryDeptId}

-- data_scope=5：仅本人数据
AND tb_biz_apply.applicant_id = #{currentUserId}
```

> 多角色处理策略：当用户拥有多个角色时，取权限范围最宽的 `data_scope` 作为最终数据权限。

### 6.3 菜单权限、数据权限、业务发起权限、流程规则的衔接

#### 菜单权限
- 控制“能否看到某菜单、能否点击某按钮”
- `permission` 同时用于前端按钮显隐和后端接口鉴权

#### 数据权限
- 控制“能看到哪些业务数据”
- 主要作用于审批单列表、业务列表、用户列表等查询类接口
- 不应用于误拦截用户自己的资料、自己的申请单、自己的待办任务

#### 业务发起权限
- 控制“能发起哪些业务类型”
- 发起权限由 `tb_biz_type_initiator` 定义，基础版按角色配置
- 流程与业务绑定，用户不需要单独拥有某个流程的发起权限

#### 流程规则
- 控制“该业务提交后走哪条流程”“流程节点最终解析给谁审批”
- 业务类型通过 `tb_biz_type.workflow_definition_id` 绑定流程定义
- 审批人由 `tb_workflow_node_approver` 定义，可按用户、角色、组织、发起人主组织主管解析

#### 统一口径
- 菜单权限决定功能入口可见性
- 数据权限决定列表结果可见范围
- 业务发起权限决定用户是否可发起某类业务
- 流程规则决定该业务如何流转、由谁审批

### 6.4 权限加载与鉴权流程

用户登录后，系统按以下顺序加载权限：

```text
用户登录
   ↓
查询用户所有角色（tb_user_role_rel）
   ↓
查询角色关联的所有菜单/按钮（tb_user_role_menu → tb_sys_menu）
   ↓
去重合并，得到用户拥有的：
  - 可访问菜单列表（type=1/2）
  - 权限标识集合（permission）
   ↓
取各角色 data_scope 最小值，得到数据权限范围
   ↓
加载用户主组织（tb_user_dept_rel.is_primary=1）
   ↓
将菜单树 + 权限标识集合 + 数据权限写入 Token/缓存
```

建议写入会话上下文或缓存的内容：
- `userId`
- `roleCodes`
- `primaryDeptId`
- `menuTree`
- `permissions`
- `dataScope`
- `dataDeptIds`

后端访问控制分两步执行：
1. 校验是否具备接口所需 `permission`
2. 若为查询接口，再叠加数据权限过滤条件

特别规则：
- 待办审批、已办审批、抄送我的任务查询，优先依据任务归属或参与关系判断
- 审批详情若当前用户是发起人、当前审批人、历史审批人之一，应允许查看必要信息

### 7.4 流转条件表 `tb_workflow_transition`

定义节点间的有向边，以及条件分支的判断表达式。

```sql
CREATE TABLE `tb_workflow_transition` (
  `id`              BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `definition_id`   BIGINT       NOT NULL               COMMENT '所属流程定义ID',
  `from_node_key`   VARCHAR(64)  NOT NULL               COMMENT '来源节点标识',
  `to_node_key`     VARCHAR(64)  NOT NULL               COMMENT '目标节点标识',
  `condition_expr`  VARCHAR(512)                        COMMENT '条件表达式（为NULL则无条件流转）',
  `priority`        INT          NOT NULL DEFAULT 0     COMMENT '条件优先级（值越小越先判断）',
  `label`           VARCHAR(64)                         COMMENT '连线标签',
  `created_at`      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `is_deleted`      TINYINT      NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  KEY `idx_definition_from` (`definition_id`, `from_node_key`)
) ENGINE=InnoDB COMMENT='流转条件表';
```

字段作用说明：
- `from_node_key`：表示从哪个节点流出
- `to_node_key`：表示流转到哪个目标节点
- `condition_expr`：表示该连线命中的条件；为 `NULL` 时表示默认路径或无条件路径
- `priority`：当同一节点存在多条出边时，按优先级从小到大依次判断
- `label`：用于流程设计器展示，例如“金额大于5000”“默认分支”

运行时作用过程：
1. 当前节点处理完成后，系统根据当前流程实例绑定的 `definition_id` 和当前节点 `node_key` 查询所有出边
2. 从 `tb_workflow_transition` 中读取 `from_node_key = 当前节点` 的所有记录
3. 按 `priority` 升序依次判断每条连线的 `condition_expr`
4. 若 `condition_expr` 为 `NULL`，表示该连线可作为无条件或兜底路径
5. 第一条满足条件的连线生效，系统激活对应的 `to_node_key` 节点
6. 若目标节点是 `END`，则流程结束；若目标节点是普通审批节点，则创建对应节点实例并继续审批
7. 若目标节点是 `PARALLEL_SPLIT`，则其所有出边都将被同时激活；若目标节点是 `PARALLEL_JOIN`，则需等待所有预期分支完成后再继续向后流转

设计理解：
- `tb_workflow_node` 负责定义“有哪些节点”
- `tb_workflow_transition` 负责定义“节点之间怎么连、满足什么条件时走哪条线”
- 因此 `tb_workflow_transition` 实际上承担了流程运行时的路由决策功能

示例：费用报销流程按金额分支

假设某报销流程 `definition_id=20` 中存在以下节点：
- `start`：开始节点
- `dept_leader`：部门负责人审批
- `finance_director`：财务总监审批
- `end`：结束节点

业务规则：
- 报销金额 `amount <= 5000` 时，部门负责人通过后直接结束
- 报销金额 `amount > 5000` 时，部门负责人通过后还需财务总监审批

则 `tb_workflow_transition` 可以配置为：

| definition_id | from_node_key | to_node_key | condition_expr | priority | label |
|---------------|---------------|-------------|----------------|----------|-------|
| 20 | `start` | `dept_leader` | `NULL` | 1 | 开始 |
| 20 | `dept_leader` | `finance_director` | `amount > 5000` | 1 | 金额大于5000 |
| 20 | `dept_leader` | `end` | `NULL` | 99 | 默认结束 |
| 20 | `finance_director` | `end` | `NULL` | 1 | 审批完成 |

示例运行过程：
- 当报销单金额为 `3000`：
  - `start` 完成后，无条件流转到 `dept_leader`
  - `dept_leader` 审批通过后，先判断 `amount > 5000`，结果为 false
  - 再命中 `condition_expr = NULL` 的默认分支
  - 流程流转到 `end`，审批结束
- 当报销单金额为 `12000`：
  - `start` 完成后，无条件流转到 `dept_leader`
  - `dept_leader` 审批通过后，判断 `amount > 5000`，结果为 true
  - 流程流转到 `finance_director`
  - `finance_director` 审批通过后，再无条件流转到 `end`

配套节点配置示例：

| definition_id | node_key | name | node_type | approve_mode |
|---------------|----------|------|-----------|--------------|
| 20 | `start` | 开始 | `START` | `NULL` |
| 20 | `dept_leader` | 部门负责人审批 | `APPROVAL` | `AND` |
| 20 | `finance_director` | 财务总监审批 | `APPROVAL` | `AND` |
| 20 | `end` | 结束 | `END` | `NULL` |

对应的审批人配置示例：

| node_key | approver_type | approver_value | 说明 |
|----------|---------------|----------------|------|
| `dept_leader` | `INITIATOR_DEPT_LEADER` | `0` 或固定占位值 | 动态解析发起人主部门主管 |
| `finance_director` | `ROLE` | 财务总监角色ID | 解析该角色下有效用户 |

ASCII 流程图示例：

```text
[START]
   |
   v
[部门负责人审批 dept_leader]
   |\
   | \ amount > 5000
   |  \
   |   v
   | [财务总监审批 finance_director]
   |          |
   |          v
   |        [END]
   |
   | 默认分支（amount <= 5000）
   v
 [END]
```

实现建议：
- 同一节点建议保留一条 `condition_expr = NULL` 的兜底分支，避免所有条件都不命中导致流程卡住
- 条件表达式所使用的变量应明确来源于 `tb_workflow_instance.form_data` 和内置上下文变量
- `priority` 应用于解决多条连线同时可能命中的判断顺序问题

## 九、工作流核心运行机制

### 9.1 工作流定义与发布

1. 管理员在设计器中配置节点、连线、审批人、条件
2. 保存为草稿（`status=0`），可反复修改
3. 发布流程（`status=1`）前自动校验：
   - 必须有且仅有一个 `START` 节点和至少一个 `END` 节点
   - 所有节点可达，不存在孤立节点
   - `APPROVAL` 节点必须配置至少一个审批人
4. 修改已发布流程时，需新建版本（递增 `version`），旧版本归档
5. 已运行实例继续使用原版本，新发起实例使用最新已发布版本

### 9.2 流程版本发布与归档机制

`tb_workflow_definition.version` 用于标识同一流程编码 `code` 下的发布快照版本。版本的核心目标是保证“流程定义可演进”与“历史实例不受影响”同时成立。

触发时机：
- 编辑草稿流程：直接修改当前草稿版本，不新增 `version`
- 编辑已发布流程：不得直接修改原已发布版本，系统需基于当前已发布版本复制出一个新草稿版本
- 发布草稿流程：执行发布校验，通过后切换生效版本

发布相关表：
- 定义层会变化的表：`tb_workflow_definition`、`tb_workflow_node`、`tb_workflow_transition`、`tb_workflow_node_approver`
- 运行层不会因发布新版本而修改的表：`tb_workflow_instance`、`tb_workflow_node_instance`、`tb_workflow_node_approver_instance`、`tb_workflow_approval_record`
- 业务单据表 `tb_biz_apply` 不因流程新版本发布而回写或迁移历史数据

编辑已发布流程时的数据变化：
1. 查询当前已发布版本，例如 `code=LEAVE_APPROVAL, version=1, status=1`
2. 在 `tb_workflow_definition` 新增一条草稿记录：
   - `code` 与原版本保持一致
   - `version = 当前最大版本号 + 1`
   - `status = 0`
3. 复制原版本对应的节点定义到新版本：
   - 复制 `tb_workflow_node`
   - 新记录的 `definition_id` 指向新版本的 `tb_workflow_definition.id`
4. 复制原版本对应的流转定义到新版本：
   - 复制 `tb_workflow_transition`
   - 新记录的 `definition_id` 指向新版本的 `tb_workflow_definition.id`
5. 复制原版本对应的审批人配置到新版本：
   - 按节点 `node_key` 建立新旧节点映射关系
   - 将旧节点下的 `tb_workflow_node_approver` 复制到新节点下
6. 后续所有设计器修改都只作用于这个新草稿版本

点击发布时的数据变化：
1. 校验当前草稿版本：
   - 必须有且仅有一个 `START` 节点
   - 至少存在一个 `END` 节点
   - 所有节点可达，不存在孤立节点
   - `APPROVAL` 节点必须存在审批人配置
2. 在同一事务内执行版本切换：
   - 将同 `code` 下原 `status=1` 的版本更新为 `status=2`
   - 将当前草稿版本从 `status=0` 更新为 `status=1`
3. 发布完成后，该 `code` 下仅保留一个已发布版本

运行中实例为什么不变：
- `tb_workflow_instance.definition_id` 在实例创建时就已绑定具体流程定义版本
- 流程实例一旦创建，后续审批节点、审批人实例、审批记录都围绕该 `definition_id` 运行
- 因此流程升级后：
  - 已运行实例继续走旧版本
  - 已完成实例保留原历史轨迹
  - 只有新发起实例才会使用最新已发布版本

与业务单据的关系：
- 业务单据提交审批时，系统先根据 `tb_biz_apply.biz_code` 查询 `tb_biz_type.workflow_definition_id`
- 创建流程实例时，将该 `workflow_definition_id` 写入 `tb_workflow_instance.definition_id`
- 若某业务类型后续改绑到新的流程定义版本，则只有改绑后新提交的单据会使用新的 `definition_id`
- 已提交单据关联的 `workflow_instance_id` 不因流程发布新版本或业务重新绑定流程而变化

示例：
- 当前已发布版本：`tb_workflow_definition(id=100, code='LEAVE_APPROVAL', version=1, status=1)`
- 管理员编辑已发布流程后，系统生成新草稿：`tb_workflow_definition(id=200, code='LEAVE_APPROVAL', version=2, status=0)`
- 发布 version=2 后：
  - version=1 更新为 `status=2`
  - version=2 更新为 `status=1`
- 此前已创建并绑定 `definition_id=100` 的流程实例继续按 version=1 运行；之后新创建的流程实例绑定 `definition_id=200`

实现建议：
- “归档”在当前表结构中用 `tb_workflow_definition.status=2` 表达
- 编辑已发布流程时，建议前端显式提示“将创建新版本草稿”
- 复制审批人配置时，建议以 `node_key` 作为新旧节点映射依据，而不是旧节点主键 ID
- 发布动作必须加事务，保证定义主表、节点、连线、审批人配置的版本切换一致

### 9.3 发起审批

前置校验：
- 根据业务类型查询 `tb_biz_type`，验证业务是否存在、是否启用，以及是否已配置 `workflow_definition_id`
- 根据业务类型查询 `tb_biz_type_initiator`，验证当前用户所在角色是否有权发起该业务
- 根据 `tb_biz_type.workflow_definition_id` 查询对应流程定义，确认流程定义存在且可用于发起

发起步骤：
1. 业务单据提交时，先根据 `tb_biz_apply.biz_code` 查询 `tb_biz_type`，获取对应的 `workflow_definition_id`
2. 在 `tb_workflow_instance` 创建实例记录（`status=RUNNING`，`definition_id=workflow_definition_id`）
3. 读取流程定义，找到 `START` 节点，激活从 `START` 出发的第一个节点
4. 激活节点时：
   - 在 `tb_workflow_node_instance` 创建记录（`status=ACTIVE`，计算 `deadline_at`）
   - 解析审批人配置，在 `tb_workflow_node_approver_instance` 为每位审批人创建记录
   - 向审批人发送待办通知
5. 在 `tb_workflow_approval_record` 写入 `SUBMIT` 操作记录

### 9.4 节点流转逻辑

```text
节点审批完成
   ↓
查询 tb_workflow_transition（from_node_key = 当前节点）
   ↓
按 priority 升序遍历出边
   ↓
评估 condition_expr（NULL 表示无条件匹配）
   ↓
第一条满足条件的边 → 激活目标节点
```

特殊处理：
- 目标节点为 `END`：将流程实例更新为 `APPROVED`
- 目标节点为 `PARALLEL_SPLIT`：同时激活所有出边对应节点
- 目标节点为 `PARALLEL_JOIN`：等待所有进边来源节点完成后才继续

### 9.5 条件分支

| 场景 | 示例表达式 |
|------|------------|
| 数值比较 | `amount > 10000` |
| 字符串匹配 | `dept_code == 'FINANCE'` |
| 逻辑组合 | `amount > 5000 && leave_days <= 3` |
| 布尔判断 | `is_urgent == true` |

运行时变量来源：
- `tb_workflow_instance.form_data` 中的字段
- 内置变量：`initiator_id`、`initiator_dept_id`、`current_time`

建议为条件节点保留一条 `condition_expr = NULL` 的兜底分支，避免流程卡死。

### 9.6 多人审批模式

#### AND（会签）
- 所有审批人必须操作完毕
- 所有人通过则节点通过
- 任意一人拒绝则节点拒绝，其余任务自动取消

#### OR（或签）
- 任意一人通过则节点立即通过，其余任务自动取消
- 所有人均拒绝则节点拒绝

#### SEQUENTIAL（顺签）
- 按 `sort_order` 升序逐一激活
- 当前人通过后激活下一人
- 任意一人拒绝则节点拒绝
- 所有人通过则节点通过

### 9.7 并行处理

并行拆分（`PARALLEL_SPLIT`）：
- 激活所有出边对应的下一个节点，多个节点同时处于 `ACTIVE`

并行聚合（`PARALLEL_JOIN`）：
- 每当一条并行分支完成后，检查所有分支状态
- 当所有预期分支均完成后，激活后续节点
- `config_json.expect_branch_count` 记录需要等待的分支数

### 9.8 超时处理

节点激活时根据 `timeout_hours` 计算 `deadline_at`，由定时任务扫描处理。

```sql
SELECT * FROM tb_workflow_node_instance
WHERE status = 'ACTIVE'
  AND deadline_at IS NOT NULL
  AND deadline_at <= NOW();
```

三种超时策略：

| 策略 | timeout_action 值 | 行为 |
|------|-------------------|------|
| 自动通过 | `AUTO_APPROVE` | 节点自动通过并继续流转 |
| 自动拒绝 | `AUTO_REJECT` | 流程自动拒绝 |
| 仅提醒 | `NOTIFY_ONLY` | 发送催办通知，继续等待人工处理 |

### 9.9 审批操作

| 操作 | 说明 | 前置条件 |
|------|------|----------|
| 通过（APPROVE） | 当前审批人同意 | 当前用户在审批人实例中且 `is_active=1` |
| 拒绝（REJECT） | 当前审批人驳回 | 同上 |
| 转交（DELEGATE） | 将审批任务转给他人 | 同上 |
| 撤回（RECALL） | 发起人撤销流程 | 流程 `status=RUNNING` |
| 加签 | 临时增加审批人 | 当前审批人在 `ACTIVE` 节点上操作 |

所有操作均写入 `tb_workflow_approval_record`。

### 9.10 消息通知

| 触发事件 | 通知对象 |
|----------|----------|
| 节点激活 | 节点审批人 |
| 节点被拒绝 | 发起人 |
| 流程完成（通过/拒绝） | 发起人 |
| 催办提醒 | 未处理的审批人 |
| 超时自动处理 | 发起人 + 审批人 |

## 十一、后台管理功能设计

### 11.1 用户管理

用户管理提供系统用户的完整生命周期管理，包括创建、修改、停用/删除，以及用户与角色、组织之间的关联维护。

核心功能：
- 新增用户
- 修改用户
- 删除用户（软删除）
- 查询用户列表
- 用户关联角色
- 用户关联组织
- 设置主组织
- 查看用户权限视图
- 重置密码
- 启停用户

关键规则：
- 校验 `username` 全局唯一
- 新增时使用 BCrypt 加密密码后写入 `tb_user`
- 修改时 `username` 不允许修改，密码通过独立接口重置
- 删除前检查是否存在进行中的审批任务
- 用户角色与组织建议采用全量替换方式维护，变更后刷新权限缓存

### 11.2 角色管理

角色管理提供角色的增删改查能力，角色是审批流权限分配和数据权限配置的基本单位。

核心功能：
- 新增角色
- 修改角色
- 删除角色（软删除）
- 查询角色列表
- 菜单授权
- 数据权限配置
- 查看角色下用户

关键规则：
- `code` 全局唯一且创建后不可修改
- 停用角色后，不再参与新的权限加载和审批人解析
- 删除角色前需先检查用户关联、业务发起权限引用和流程规则引用

### 11.3 组织管理

组织管理提供树形部门的完整维护能力，是审批流中“按组织分配审批人”和“动态解析组织主管”的基础。

核心功能：
- 新增部门
- 修改部门
- 移动部门
- 删除部门（软删除）
- 查询部门树
- 维护部门主管
- 查看部门用户

关键规则：
- 移动部门时需重新计算 `path` 和 `level`
- 删除部门前需检查无子部门、无用户关联、无关键流程引用
- 部门主管应为有效用户，且通常属于该部门

### 11.4 菜单管理

菜单管理提供系统菜单和按钮权限的完整维护能力，是 RBAC 模型中“功能权限”的数据来源。

核心功能：
- 新增菜单/按钮
- 修改菜单/按钮
- 删除菜单/按钮
- 查询菜单树
- 为角色分配菜单/按钮
- 查询角色已有权限

关键规则：
- 按钮节点必须挂在菜单节点下
- 按钮类型必须配置 `permission`
- `permission` 应保持全局唯一
- 隐藏菜单不影响接口访问权限，只影响导航展示

### 11.5 业务类型管理

业务类型管理维护系统支持的审批业务种类，是流程定义、业务申请和业务发起权限的基础配置。

核心功能：
- 新增业务类型
- 修改业务类型
- 删除业务类型（软删除）
- 查询业务类型列表
- 维护业务名称和业务说明
- 配置默认流程定义
- 配置业务发起权限
- 查询业务已配置发起角色

关键规则：
- `bizCode` 全局唯一且创建后不可修改
- 业务类型通过 `workflow_definition_id` 直接绑定流程定义
- `workflow_definition_id` 必须指向有效流程定义，且应绑定到当前业务口径对应的流程
- `tb_biz_apply.form_data` 仅保存业务申请数据快照，不再依赖独立业务配置明细表
- 发起权限按业务类型配置在 `tb_biz_type_initiator` 中，不再按流程单独配置
- 删除前需检查是否已被 `tb_biz_apply.biz_code` 或 `tb_biz_type_initiator.biz_code` 引用，并确认业务停用后再处理

### 11.6 字典管理

字典管理维护审批申请中可复用的枚举型基础数据，包括字典类型和字典项，是 `tb_biz_apply.form_data` 中枚举字段值的统一来源。

核心功能：
- 新增字典类型
- 修改字典类型
- 删除字典类型（软删除）
- 查询字典类型列表
- 新增字典项
- 修改字典项
- 删除字典项（软删除）
- 查询字典项列表
- 启用/停用字典类型和字典项

关键规则：
- `type_code` 全局唯一且创建后不可修改
- 同一 `type_code` 下 `item_value` 唯一
- 停用字典项后，不再允许新表单选择，但不回写历史单据中的 `form_data`
- 删除字典类型前需检查是否仍存在未删除的字典项，删除字典项前需检查是否仍被表单配置引用

## 十二、接口设计概要

> 接口统一前缀：`/sys`、`/workflow`
> 请求/响应格式：JSON
> 认证方式：认证服务端签发 JWT，资源服务端校验 JWT
> 接口方法约定：查询统一使用 `GET`，新增、修改、删除、启停、授权、审批处理等非查询操作统一使用 `POST`
> 参数传递约定：`POST` 请求参数统一使用 JSON 请求体；`GET` 请求使用路径参数或查询参数
> `Controller` 层入参统一使用 `QTO`，出参统一使用 `VO`
> 所有接口统一返回 `Resp<T>`，标准结构为 `code`、`msg`、`data`
> 所有分页查询必须同时提供对应列表查询接口，且两者查询条件、查询 SQL、返回字段、排序规则保持一致；唯一差异为分页接口额外包含分页参数和分页结果

- 两者运行在同一个 Jar 中，但通过独立安全过滤链划分职责
- 认证服务端负责签发 Token、撤销 Token、提供 Token 内省能力
- 资源服务端负责校验 Bearer Token 并保护业务接口

当前阶段认证实现说明：
- 认证框架统一使用 `Spring Security`
- 认证模型统一使用“认证服务端 + 资源服务端”
- 登录态统一使用无状态 `opaque Bearer Token`
- 请求头统一传递：`Authorization: Bearer {token}`
- 认证服务端发行方统一通过 `workflow.security.issuer` 配置
- Token 有效期统一通过 `workflow.security.token-expire-seconds` 配置
- OAuth2 客户端统一通过 `workflow.security.client-id` 与 `workflow.security.client-secret` 配置
- 用户名密码登录统一通过 `/oauth2/token` 的自定义 `grant_type=password_login` 完成
- 令牌签发、刷新、撤销统一优先走认证服务端标准端点，禁止优先扩展平行自定义登录发 token 接口
- 为满足“撤销后立即失效”，资源服务端统一基于令牌内省口径判断 token 是否有效

标准交互建议：
- 客户端通过 `/oauth2/token` 提交 `grant_type=password_login`、`username`、`password` 申请访问令牌
- 已签发令牌如需撤销，统一调用 `/oauth2/revoke`
- 外部资源服务如需独立校验 token，可调用 `/oauth2/introspect`
- 业务接口访问完成鉴权后，可通过 `/sys/auth/current-user` 获取当前登录用户上下文

初始化 SQL 默认提供管理员账号：
- 用户名：`admin`
- 密码：`admin123`

## 十三、与审批工作流的衔接规则

### 13.1 发起权限衔接

- 流程与业务类型通过 `tb_biz_type.workflow_definition_id` 绑定
- 发起权限配置在 `tb_biz_type_initiator`，按业务类型控制可发起角色范围
- `tb_biz_type_initiator` 采用“一条记录对应一个角色”的方式存储，不在单条记录中混存多个角色 ID
- 用户发起审批时，系统同时检查：
  - 用户状态正常
  - 当前业务类型存在且状态正常
  - 当前业务类型已配置有效的 `workflow_definition_id`
  - 当前用户至少存在一个有效角色命中该业务的发起配置
  - 若业务单据要求记录部门归属，则取主组织作为默认 `dept_id`

### 13.2 审批人解析衔接

- `tb_workflow_node_approver.approver_type=USER`：直接定位用户
- `tb_workflow_node_approver.approver_type=ROLE`：按角色找出有效用户集合
- `tb_workflow_node_approver.approver_type=DEPT`：按组织找出候选审批人或组织主管
- `tb_workflow_node_approver.approver_type=INITIATOR_DEPT_LEADER`：按发起人主组织的 `leader_id` 解析

### 13.3 变更影响检查

- 用户停用时：检查是否存在待办审批、是否担任部门主管、是否被直接配置为审批人
- 角色停用时：检查是否被 `tb_biz_type_initiator`、`tb_workflow_node_approver` 引用
- 组织停用或移动时：检查是否影响业务发起权限配置、审批人解析路径、数据权限口径
- 菜单调整时：仅影响后台功能入口与接口访问，不应直接改变历史流程实例数据

### 13.4 审批任务与数据权限边界

- 数据权限用于“查哪些业务数据”
- 审批任务用于“我是否是任务参与者”
- 若某审批任务已分配给当前用户，则即使该单据不在其常规数据权限范围内，也应允许其处理任务和查看必要上下文

### 13.5 业务申请数据衔接

- 用户发起业务前，系统先按 `biz_code` 确认业务类型是否启用、当前用户角色是否具备发起权限，以及 `tb_biz_type.workflow_definition_id` 是否已配置
- 页面表单可按业务类型分别实现，提交时统一写入 `tb_biz_apply.form_data`
- 用户提交后，后端按业务类型做基础参数校验；校验通过后先读取 `workflow_definition_id`，再进入流程发起
- `tb_biz_apply.form_data` 作为业务数据快照保存，不直接依赖独立业务表结构配置

### 13.6 申请字典衔接

- `tb_biz_apply.form_data` 中的枚举字段统一使用字典项值存储，不直接存储字典显示名称
- 前端渲染页面时，可根据业务场景按需加载对应字典类型的数据作为候选值
- 表单提交时，仅校验提交值是否为指定字典类型下的有效 `item_value`
- 审批流转过程中，`form_data` 作为业务快照参与条件判断和展示，历史单据不因字典文案调整而变更
- `tb_biz_type` 负责定义业务类型，字典表负责定义业务中可复用的枚举项，两者共同支撑业务申请和审批流转

## 十四、边界条件与校验规则

### 14.1 用户相关

- `username` 全局唯一
- 删除用户前必须校验无进行中待办任务
- 停用用户后不可登录，不可被新流程解析为审批人
- 用户至少应保留一个有效组织关联后再允许参与流程发起

### 14.2 角色相关

- `code` 全局唯一且创建后不可修改
- 停用角色后，不再参与新的权限加载和审批人解析
- 删除角色前需先移除用户关联，并建议同时检查业务发起权限和流程规则引用

### 14.3 组织相关

- 同一用户只能有一个主组织
- 部门树不得形成循环引用
- 部门删除前必须无子部门、无用户关联、无关键流程引用
- 部门主管应为有效用户，且通常属于该部门

### 14.4 菜单相关

- 按钮节点必须挂载在菜单节点下
- 按钮类型必须配置 `permission`
- 目录可无 `permission`，菜单建议配置列表访问类 `permission`
- `permission` 应保持全局唯一，避免鉴权歧义

### 14.5 授权相关

- 用户角色分配、用户组织分配、角色菜单授权、角色数据权限配置均建议采用全量替换
- 业务发起权限配置建议采用全量替换
- 所有授权变更应在事务中完成
- 授权完成后应刷新用户权限缓存或使旧缓存失效

### 14.6 业务申请相关

- `biz_code` 必须与 `tb_biz_type.biz_code` 保持一致，且业务停用后不可再发起新单据
- `tb_biz_type.workflow_definition_id` 不能为空，且必须指向有效流程定义；未配置流程时不允许提交审批
- `tb_biz_apply.form_data` 中的字段由具体业务场景自行约定，后端至少应校验必需参数是否齐全
- 同一业务类型的申请数据结构允许逐步演进，但历史单据按已存快照展示，不强制回写旧数据
- 业务类型改绑新的 `workflow_definition_id` 后，仅影响新提交单据，已创建实例继续沿用原 `definition_id`
- 删除业务类型前，需检查是否已存在历史业务申请或业务发起权限引用

### 14.7 字典相关

- `type_code` 必须全局唯一，`item_value` 必须在同一 `type_code` 下唯一
- 字典类型或字典项停用后，不再用于新建和编辑申请的候选项校验，但历史单据中的既有值仍允许展示
- 删除字典类型前，必须先删除或停用其下全部字典项
- 删除字典项前，若已有历史业务单据使用该值，建议优先停用而非直接删除
- 业务提交时，若枚举字段传入的 `item_value` 不属于指定字典类型，应直接校验失败
- 字典显示文案 `item_label` 可调整，但不得影响历史 `form_data` 中已保存的业务值口径

## 十五、注意事项与扩展点

### 15.1 注意事项

1. 并发控制：审批操作需加分布式锁或乐观锁，防止并发重复处理同一节点
2. 事务一致性：节点状态变更、实例状态变更、操作记录写入必须在同一事务中完成
3. 版本隔离：流程实例创建后绑定具体 `definition_id`，不随流程升级受影响
4. 表单数据安全：`form_data` 存储快照，敏感字段建议加密处理
5. 条件表达式执行需做安全控制，避免注入与非法函数调用

### 15.2 扩展点

- 动态审批人：支持在表单中直接指定审批人（如后续扩展 `FORM_FIELD`）
- 会签比例：支持“N人中M人通过即通过”
- 子流程：审批节点支持嵌套调用另一个流程定义
- 消息模板：通知内容可按流程类型配置不同模板
- 接口级权限：后续可独立抽象 `tb_sys_permission`、`sys_role_permission`
- 审批人配置范式化：后续可将 `tb_workflow_node_approver` 演进为“一条审批主体一条记录”的更规范模型

## 十六、开发落地建议

- 优先以本文件作为单文档闭环，不再拆出第二套系统权限设计文档
- 开发表结构时直接复用本文件中 `biz_*`、`user_*`、`sys_*` 和 `workflow_*` 命名，避免并行体系
- 接口设计优先围绕“用户、角色、组织、菜单、认证上下文、业务类型、工作流”七个模块展开
- 发起入口优先按业务类型建设，不暴露“用户先选流程再发起”的交互模式
- 权限中台先实现当前设计，待出现接口级细粒度授权需求后，再演进到独立权限点模型
