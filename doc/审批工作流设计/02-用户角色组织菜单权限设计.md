# 用户角色组织菜单权限设计

## 1. 设计范围

本文件聚焦系统基础权限域设计，包括用户、角色、组织、菜单、按钮、数据权限、认证授权和当前登录上下文。

## 2. 设计目标

- 统一用户、角色、组织、菜单和权限模型。
- 为后台管理、接口鉴权、数据权限和审批人解析提供稳定基础。
- 保证“认证服务端 + 资源服务端”在同一个 Jar 内的职责清晰。

## 3. 核心模型

### 3.1 用户

核心表：
- `tb_user`

职责：
- 存储账号、密码、状态、基础资料。
- 作为认证登录主体。

约束：
- `username` 全局唯一。
- 停用后不可登录，不可被新流程解析为审批人。
- 不直接固化角色外键和组织外键。

### 3.2 角色

核心表：
- `tb_user_role`
- `tb_user_role_rel`

职责：
- 聚合菜单权限、按钮权限、数据权限、业务发起权限语义。

约束：
- 一个用户可拥有多个角色。
- `code` 全局唯一且创建后不可修改。
- 停用后不再参与新的权限加载和审批人解析。

### 3.3 组织

核心表：
- `tb_user_dept`
- `tb_user_dept_rel`

职责：
- 维护组织树、主组织、主管关系、层级结构。

约束：
- 一个用户可关联多个组织。
- 一个用户有且仅有一个主组织。
- 组织树不得出现循环引用。

### 3.4 菜单与按钮

核心表：
- `tb_sys_menu`
- `tb_user_role_menu`

职责：
- 表达目录、菜单、按钮和接口权限点。

约束：
- 按钮必须挂在菜单下。
- 按钮必须配置唯一 `permission`。
- 隐藏菜单只影响导航展示，不直接改变接口鉴权。

## 4. 数据权限设计

核心表：
- `tb_user_role`
- `tb_user_role_dept`

`data_scope` 建议枚举：

| 值 | 名称 | 说明 |
|----|------|------|
| `1` | 全部数据 | 查看系统内所有数据 |
| `2` | 自定义部门 | 查看指定部门数据 |
| `3` | 本部门及子部门 | 查看主部门及子树数据 |
| `4` | 仅本部门 | 查看主部门数据 |
| `5` | 仅本人数据 | 仅查看本人数据 |

规则：
- 用户拥有多个角色时，最终数据权限取范围最宽的一档。
- 数据权限主要作用于查询类接口。
- 待办、已办、我的发起等任务视图优先按任务参与关系判断，不应被普通数据权限误拦截。

## 5. 菜单权限、数据权限与业务权限衔接

- 菜单权限控制功能入口是否可见。
- 按钮 `permission` 控制接口操作权限。
- 数据权限控制查询结果范围。
- 业务发起权限控制用户能发起哪些业务。
- 流程规则控制审批单如何流转以及由谁审批。

## 6. 后台功能设计

### 6.1 用户管理

核心功能：
- 新增、修改、删除用户
- 用户列表、分页、详情
- 重置密码
- 启停用户
- 用户角色全量分配
- 用户组织全量分配

关键规则：
- 删除前检查是否存在进行中的审批任务。
- 新增时密码统一加密。
- 用户角色与组织变更后需要刷新权限上下文。

### 6.2 角色管理

核心功能：
- 新增、修改、删除角色
- 角色列表、分页、详情
- 菜单授权
- 数据权限配置
- 查看角色下用户

关键规则：
- 删除前检查用户关联、业务发起权限引用和流程规则引用。

### 6.3 组织管理

核心功能：
- 新增、修改、移动、删除部门
- 部门树、详情
- 维护部门主管
- 查看部门下用户

关键规则：
- 删除前检查无子部门、无用户关联、无关键流程引用。
- 移动部门时需重算 `path` 和 `level`。

### 6.4 菜单管理

核心功能：
- 新增、修改、删除菜单和按钮
- 查询菜单树
- 角色菜单授权

关键规则：
- `permission` 必须保持全局唯一。

## 7. 认证与授权设计

当前项目采用“认证服务端 + 资源服务端”同进程部署：

- 认证服务端负责 `/oauth2/token`、`/oauth2/revoke`、`/oauth2/introspect`
- 资源服务端负责 Bearer Token 校验与业务接口保护
- 两者部署在同一个 Jar 中，但通过独立安全过滤链划分职责

Token 策略：
- 登录态统一使用 `opaque token`
- 资源服务端基于内省口径判断令牌有效性
- 用于满足“撤销后立即失效”

用户名密码登录方式：
- 统一通过 `/oauth2/token` 扩展自定义 `grant_type=password_login`
- 不新增平行的自定义发 token 登录接口

统一配置项：
- `workflow.security.issuer`
- `workflow.security.token-expire-seconds`
- `workflow.security.client-id`
- `workflow.security.client-secret`

## 8. 当前登录上下文

建议上下文包含：
- `userId`
- `roleCodes`
- `primaryDeptId`
- `permissions`
- `dataScope`
- `dataDeptIds`

登录后建议加载顺序：
1. 查询用户角色。
2. 查询角色授权菜单与按钮。
3. 汇总权限标识与菜单树。
4. 计算最终数据权限。
5. 加载主组织。
6. 写入上下文或授权体系可读取位置。

## 9. 主要接口范围

用户相关：
- `/sys/user/*`

角色相关：
- `/sys/role/*`

组织相关：
- `/sys/dept/*`

菜单相关：
- `/sys/menu/*`

认证相关：
- `/oauth2/token`
- `/oauth2/revoke`
- `/oauth2/introspect`
- `/sys/auth/current-user`

## 10. 边界规则

- 数据权限用于“查哪些数据”，不是“是否拥有审批任务”。
- 用户停用、角色停用、组织调整都需要评估对审批解析和数据权限的影响。
- 所有授权类修改建议采用全量替换，并在事务中完成。
- 授权完成后应刷新缓存或使旧权限上下文失效。
